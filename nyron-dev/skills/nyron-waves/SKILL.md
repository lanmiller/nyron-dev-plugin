# Nyron Waves — волновая оркестрация AI Evolve через Jira

Скилл переводит текущую сессию в режим **диспетчера**. С этого момента и до конца
работы сессия — оркестратор и фасилитатор, а не исполнитель. Центральная шина
всего — **Jira** (проект DEV, nyron.atlassian.net): брифы, статусы, отчёты,
доработки живут в тикетах; git — только для кода.

## Локальная конфигурация (определить в первый ход)

Скилл общий для команды — привязок к конкретной машине нет. В первый ход
диспетчер читает **конфиг проекта** (скилл `project-config`:
`.claude/nyron-dev.md` + `.local.md`) — он переопределяет хранилище задач
(jira/files), карту моделей и тип рабочих (код/данные); конфига нет — дефолты
ai-evolve ниже. Затем определяет и фиксирует для себя две вещи:

- **`<PROJECT_ROOT>`** — корень чекаута ai-evolve на ЭТОЙ машине: каталог, где
  лежат подпапки `ai-evolve-front/`, `ai-evolve-back/`, `n8n/`,
  `ai-evolve-docs-test/`, `canvas-mcp-server/`. Обычно это cwd сессии; если
  нет — найти и явно зафиксировать. Все пути ниже и в брифах — от него, в чипы
  подставлять УЖЕ РАЗВЁРНУТЫЙ абсолютный путь (сессия-волна не знает плейсхолдеров).
- **Git — только по SSH-ключам**, никаких токенов в скилле. MR-ссылку git сам
  печатает при пуше ветки («To create a merge request… visit: …» — волна кладёт
  её в отчёт-коммент), ревью идёт по `git log main..<ветка>`, CI проверяется
  смоуком predprod по факту.

## Якоря проекта (читать как аксиомы)

- **Рабочий каталог всегда `<PROJECT_ROOT>`** — и у диспетчера, и у волн.
  Корень — НЕ git; git-репо — подпапки `ai-evolve-front/`, `ai-evolve-back/`,
  `n8n/`, `ai-evolve-docs-test/`, `canvas-mcp-server/`.
- **Пуш в main front/back/n8n = авто-CI на predprod** (predprod.evolve.ru).
  Predprod — полигон, катать можно, НО: (а) волны в main НЕ пушат — финал волны
  всегда ветка + MR/PR-ссылка в тикете; (б) мерж пачкой делает только диспетчер
  и **никогда во время live-прогона пользователя** (пуш = HMR/перезагрузка его
  стенда). Ветки `prod2` и `nyron` не трогать вообще — отдельные релизные MR.
- **Jira — единственный канал коммуникации волн**: бриф волны = коммент
  диспетчера в тикете; отчёт волны = коммент в тикете; вердикт диспетчера =
  коммент в тикете. MCP-тулзы atlassian доступны всем сессиям без подтверждений —
  поэтому шина работает автономно (в отличие от cross-session сообщений,
  требующих ручного клика).
- **Jira = полная событийная шина** (тезис плагина, обкатан двусторонне):
  комменты — сообщения, статусы — конечный автомат, переходы — события, на
  которые сессии просыпаются вотчерами за ~минуту. Следствие: человек управляет
  системой перетаскиванием карточек (хоть с телефона) — тикет в «Доработка» +
  коммент = команда волне, без терминала. История статусов и комментов =
  бесплатный аудит-лог операционки.
- **Jira-сервер — из конфига проекта** (`jira.mcp_server`, канон всех
  проектов — локальный project-MCP в `.mcp.json` корня: докерный
  `mcp-atlassian` по API-токену, headless-надёжен; в ai-evolve это
  `jira-nyron`, тулзы `mcp__jira-nyron__jira_*`; настройка — скилл
  `project-config`). Первым ходом (и диспетчер, и волны) — смоук
  `jira_get_user_profile`/JQL по проекту. Fallback, если project-MCP
  недоступен (Docker лежит): плагинный Atlassian MCP (`plugin:atlassian`)
  С ГРАНТОМ на сайт проекта — проверять грант через
  `getAccessibleAtlassianResources`; чужие Atlassian-коннекторы профиля без
  гранта не годятся.
- **Статус-машина DEV** (реальные статусы воркфлоу): Бэклог/К выполнению →
  В работе → **Готово к тестированию (Ревью)** → (при замечаниях →
  **Доработка** → снова Ревью) → Тестирование → Готово; служебные: Пауза,
  Дизайн; Отменено — только для дублей. Волна сдаёт тикет в «Готово к
  тестированию (Ревью)»; вердикт «ДОРАБОТКА» диспетчера → статус «Доработка»;
  «Тестирование» — стадия проверки постановщиком/QA после ревью. Имена статусов
  могут отличаться между досками — ориентироваться на категорию
  (new / indeterminate / done), transition id получать
  `getTransitionsForJiraIssue` по месту (известные на эпик-доске: Готово=41,
  Ревью=4, Доработка=3, Тестирование=5, В работе=2, Бэклог=11, Отменено=6).
  Типы: задачи — «Задание» (НЕ «Задача»), баги — «Баг», подзадачи — «Subtask».
  Лейблы: front / back / agent / infra / catalog.
- **Стандарт задачи** — отдельный скилл `jira-task-standard` этого плагина
  (JTBD, MECE, McKinsey-стиль, два слоя, DoD, блок «Как тестировать», шаблоны).
  Каждый тикет, входящий в волну, обязан ему соответствовать; доведение чужих
  и бизнесовых тикетов до стандарта — на груминге, не отбрасыванием.
- **CI predprod идёт с main автоматически** — отдельного управления не требует.
- **Секреты и доступы** (Langfuse-ключи, SSH-алиасы серверов) — из локального
  CLAUDE.md/памяти конкретной машины; в брифы и чипы секреты не вписывать.
- **Канон правил волн — `ai-evolve-docs-test/shared/wave-rules.md`** (docs-репо,
  пушится безопасно). Если файла нет — восстановить из
  `assets/wave-rules-template.md` этого скилла и запушить docs-test.
- **Папка эпика — `ai-evolve-docs-test/waves/<DEV-эпик>-<слаг>/`**: всё, что
  длиннее коммента (спеки, ресёрчи, планы волн, дампы, длинные отчёты), кладётся
  файлами сюда, а в тикет — ссылка на файл. Папка — ПОД ЭПИК, не под задачу
  (иначе зоопарк). Jira = статусы и шина; папка = вложения. Короткое (брифы,
  отчёты, вердикты) остаётся комментами.
- **Язык — русский** везде: брифы, чипы, комменты, коммиты.

## Роль сессии-диспетчера

Диспетчер делает ТОЛЬКО это:
- **груминг с пользователем**: выбрать эпик (или пачку задач), пройтись по
  тикетам — что делаем, что нет, что дубль, что переносим; итоги груминга
  зафиксировать в тикетах (комменты/лейблы/статусы) ДО запуска волн;
- нарезка волн по участкам кода (тикеты одной волны = один участок, чтобы волны
  не толкались в файлах) и брифы-комменты в тикеты. **Каждый бриф кодового
  тикета несёт «Паспорт конвейера»** (JSON-блок по шаблону брифа) — готовые
  args обоих workflow; копия в чип. **Критерий готовности тикета в волну:
  паспорт собирается без вопросов** (DoD не превращается в список / behavioral
  неясен → тикет в волну не идёт, дорабатывается постановка). Волна первым
  ходом запускает WF-тестген с паспортом, а не переваривает прозу — «руками
  доделаю» побеждает ровно там, где вход не переварен грумингом.
  **Не нарезать на вырост:**
  в задачу не закладывать абстракции и опции «под будущее», которых нет в
  требовании — исполнитель честно построит лишнее, и лестница до кода его не
  спасёт, потому что «так было поставлено»;
- формирование и выдача /goal-чипов (spawn_task): клик по чипу открывает
  отдельную сессию-волну, текст чипа — её цель;
- мониторинг: гибрид вотчеров — фоновые until-loop'ы на git- и Jira-события
  (пробуждение за ~1–2 мин) + JQL по тикетам волн ПЕРВЫМ действием на каждом
  пробуждении; ScheduleWakeup — только страховочный пульс
  (механика — `references/wave-protocol.md`);
- ревью: **кросс-ревью другой моделью** (скилл `cross-review`: GPT через
  codex — «кто писал — тот не проверяет»; вердикт волна прикладывает к отчёту,
  диспетчер сверяет/перегоняет) + при надобности code-reviewer-субагент (opus);
  результат — вердикт-комментом в тикет (`ДОРАБОТКА: …` / `ПРИНЯТО`);
  перевод статусов. **Вердикт диспетчера = свой взгляд на дифф**, не резиновая
  печать поверх codex-вердикта: перед «ПРИНЯТО» диспетчер смотрит сам дифф и
  отчёт, а не только вердикты. Тикет помечен в брифе как **сложный**
  (архитектура, миграция, конструкция в целом) → перед вердиктом точечный
  разбор конструкции на дорогой модели (`models.thinker`) — codex ловит
  построчно, «конструкция неправильная целиком» — его слепое пятно;
- сведение: порядок мержа, конфликты, финальные проверки, сводка пользователю;
  мерж пачкой в main — только после явного «ок» пользователя и вне live-прогонов;
- после мержа: смоук predprod, тикеты волны → Готово (с комментом-пруфом),
  **реиндекс графа кода** смерженного репо, если в проекте настроен
  codebase-memory (`index_repository`, секунды — иначе граф врёт про main),
  чипы следующих волн.

Диспетчер НЕ делает сам: не пишет код, не ресёрчит, не гоняет массовые операции.
Контекст главной сессии — самый дорогой ресурс: он для целостной картины эпика,
очереди волн и приёма отчётов. Esc в главной сессии убивает её фоновых
субагентов безвозвратно — вся работа уходит в отдельные сессии.

### Исключения: когда диспетчер трогает код сам

1. **Мелочь на ревью MR** — опечатки, стиль, точечный фикс прямо в ветке волны.
   Граница: если нужно менять логику — возвращать волне вердиктом.
2. **Эскалация** — волна 2–3 итерации подряд ошибается в одном месте.
   Вмешательство точечное, причина — комментом в тикет.

## Роутинг моделей

Модель указывается ЯВНО в каждом вызове субагента и в каждом чипе — наследование
запрещено. Дефолты ниже; конфиг проекта (`models.*`) переопределяет.

**Принцип: дорогая модель — на решениях, механика — в Workflow на дешёвых.**
Сессия (диспетчер, волна) касается работы только в точках решений: бриф,
чекпойнт, вердикт, блокер. Всю механику между точками гоняет Workflow-скрипт
стадиями на дешёвых моделях. Сессия, тянущая механику руками, — анти-паттерн.

| Модель | Роль |
|---|---|
| **Fable** | Решения: груминг, брифы, чекпойнты конвейера, вердикты, разбор сложных конструкций, приёмка пути пользователя. Целевая модель и диспетчера, И волн (`models.dispatcher`/`models.wave`) — но ТОЛЬКО при конвейере-Workflow (иначе сжигает бюджет механикой). Код не пишет. |
| **Opus** | Имплементация: код, диагностика, адверсарное ревью, сложный тест-дизайн. |
| **Sonnet** | Обработка, поиск, ресёрч, черновики, типовые тесты (рутина — effort low). |
| **Haiku** | Массовые однотипные операции, механические валидаторы. |
| **Sol** (codex) | Кросс-ревью, план-челлендж, тест-ген из DoD — на подписке, Claude-токены не ест. |

**Одна сессия — одна модель на всю жизнь**: переключение модели внутри сессии
обнуляет KV-кэш и заново греет контекст (плюс человеческий фактор «забыл
переключить»). Разнообразие моделей — только через субагентов и Workflow-стадии.

Внутри волны: **механика → Workflow, решения и ДЛИННОЕ → сессия** (конвейер
v2.1, гибрид по итогам пилота НБ-1). Жёсткое разделение по природе шага:

- **Workflow** — короткая детерминированная механика: код-стадии, фикс-итерации,
  дифф-валидатор, адверсарка, телеметрия, fan-out пачек. Один workflow-агент
  не может ждать внешний процесс дольше Bash-лимита (10 мин) и не умеет
  «повиснуть до колбэка» — НЕ пытаться.
- **Сессия** — решения (чекпойнты, вердикты, блокеры, Jira) и **длинные
  внешние процессы**: codex CLI, docker-сьюты — запускаются
  `run_in_background: true`, сессия занимается другим, завершение будит её
  task-notification'ом. **Sleep-поллы запрещены** — нотификация уже есть
  (грабля НБ-1: волна спала слепыми 9-минутными поллами).

**Конвейер тикета = ДВА workflow + сессионные интерлюдии:**

1. **WF-тестген**: task-файл из DoD → codex-запуск готовится, длинный прогон
   codex сессия ведёт фоном → red-прогон → возврат тестов и red-статуса.
   (Мелкий тикет — тест-ген можно целиком сессией фоном, без WF.)
2. **Чекпойнт сессии**: тесты = спека; здесь же можно уточнить задание кодеру
   (на пилоте после чекпойнта уточнялся стенд прогона) — вход WF-импл =
   одобренные тесты + уточнённый бриф.
3. **WF-импл**: код (red→green, тестовые файлы не трогает) → дифф-валидатор
   (haiku) → **адверсарка ТОЛЬКО на метке СЛОЖНЫЙ** → фикс-итерации с лимитом.
   Канон по данным двух пилотов psylia: простой тикет (НБ-1) — адверсарка
   0 уникальных, всё дубли Sol; сложный сквозной (НБ-2) — 12 находок за
   4 раунда, и Sol ПОСЛЕ неё добыл ещё 7 своих — слои ДОПОЛНЯЮТ друг друга,
   не дублируют. Итог: обычный тикет → Sol один; СЛОЖНЫЙ → оба слоя.
4. **Ревью-циклы Sol** (сессия, фоном): **r1 — полный; r2+ — по диффу фиксов;
   раунд без НОВЫХ находок → СТОП** (на НБ-1 r3 дал ноль нового и сжёг 13 мин
   из 66 — ~20% wall-clock). Больше половины находок r2 — рябь от фиксов r1:
   ре-ревью диффа достаточно.
5. **Полный сьют — ОДИН раз, после последней фикс-итерации** (не параллельно
   фиксам — прогон на живых маунтах бракуется). Доработка после вердикта с
   чисто ТЕСТОВЫМ диффом перегона полного сьюта не требует.
6. **Вердикт сессии-волны → сдача**: `hub_post` с MR (канон 0.6.2), отчёт,
   телеметрия complete.

Внутри стадий — проектные субагенты (front-dev / backend-dev / langgraph-dev /
code-reviewer, обязательны по CLAUDE.md; модель явно). Сама волна руками код
не пишет. Скелет — `examples/ticket-conveyor-workflow.md`.

## Телеметрия конвейера (чем доказываем, что практики работают)

Каждая стадия конвейера дописывает JSONL-строку в `.nyron-hub/metrics.jsonl`
(append, тот же якорь, что будка): `{ts, ticket, wave, stage, source, type,
data}`. Обязательные события: пруф лестницы №2 (`source: graph|grep|none`),
каждая находка ревью (`source: sol|adversarial|red-test|dispatcher`,
`type: bug|overengineering|dup|style`), вердикты с номером итерации, red→green
переходы. Сводка волны — по этому файлу, не по воспоминаниям.

Зачем: решения о конвейере принимаются по цифрам, не по ощущениям. Так решён
вопрос адверсарки (два пилота → «только СЛОЖНЫЕ», см. выше); тем же способом
решается судьба любого нового слоя. LOC —
не метрика (урок ponytail); метрики — возвраты на доработку, уникальные
находки по источникам, время бриф→ПРИНЯТО.

**Правило саморедукции:** слой проверки, не давший УНИКАЛЬНЫХ находок за
2 волны подряд (по телеметрии), вылетает или сужается до метки СЛОЖНЫЙ.
Распространяется на все слои, включая будущие «а давайте ещё проверим»:
каждый прогон/тест/токен обязан называть класс ошибок, который ловит только
он; не может назвать — слой лишний. Тест-бюджет: 1 пункт DoD = 1-2 теста
+ негативные, MECE; тест без пункта DoD, который он доказывает, — лишний код.

**НЕБЛОКИРУЮЩЕЕ не теряется**: финальная стадия конвейера волны собирает
неблокирующие замечания (включая «переусложнение» из cross-review п.7) и
дописывает в файл долгов проекта — тикеты не шумят, мины лежат в одном месте.

## Механизм: чип → сессия-волна → Jira-тикеты

1. Диспетчер пишет **бриф-коммент** в каждый тикет волны (шаблон —
   `assets/wave-brief-template.md`): диагноз, файлы/участок, инварианты,
   критерии финала, имя ветки. Полный бриф живёт в Jira, НЕ в чипе.
2. Создаёт **suggested task** (spawn_task). Чип начинается со slash-команды
   `/goal` (НЕ `goal:` текстом), лимит 4000 символов — поэтому чип короткий
   указатель: «исполни волну N: тикеты DEV-X, DEV-Y — прочитай бриф-комменты
   диспетчера в них и правила `ai-evolve-docs-test/shared/wave-rules.md`».
3. Волна первым делом читает тикеты + правила, переводит тикеты **В работу**,
   фиксирует коротким комментом свой план (самопроверка брифа). Бриф
   противоречив — не работать наугад: коммент-блокер и стоп.
4. Работает по правилам волн; финал: ветка + MR, **отчёт-коммент** в каждый
   тикет (что сделано, ссылка на MR/коммиты, чем проверено, что НЕ сделано),
   тикет → **Тестирование**, assignee → постановщик. После этого волна НЕ
   завершается — ждёт вердикт диспетчера в комментах тикета (шина ревью).

Полный протокол (формат чипа, шина ревью, guard'ы зависимых волн, мониторинг) —
`references/wave-protocol.md`. **Прочитать перед первым запуском обязательно.**

## Автономный режим (fire-and-forget)

- **Одно подтверждение**: план волн (очередь, зависимости, брифы, чипы)
  показывается пользователю целиком, после «да» вопросов больше нет; развилки
  диспетчер решает сам и фиксирует комментами в тикетах.
- **Все чипы — сразу**, включая зависимые волны (им в чип вшит guard).
- **Не блокироваться на человеке**: основной поток — Jira-комменты и MR;
  cross-session сообщения — только дублирующий канал.
- **Границы автономности**: мерж в main — только с явного «ок» пользователя
  (и вне его live-прогонов); тикеты в «Готово» — только после мержа/смоука,
  до того — «Тестирование».
- **Настоящий блокер** — коммент в тикет + статус, ждать.

## Лестница до кода (кодовые волны)

Самый дешёвый код — ненаписанный. В бриф КАЖДОЙ кодовой волны (`type:
development`) идут три вопроса, на которые исполнитель отвечает ДО первой
строки, а ответ кладёт в отчёт: (1) нужен ли этот код вообще; (2) нет ли уже в
кодовой базе — **с пруфом**: в проекте настроен граф кода (codebase-memory —
`search_graph`/`semantic_query`) → пруф через граф, это надёжнее; нет графа
или индекс стух → grep; «вроде нет» пруфом не считается; (3) нельзя ли
параметризацией существующего. Это наш канон «дорабатывай легаси» в виде
обязательного шага. Граф недоступен — фолбэк на grep, не блокер.

**Ограничитель обязателен.** Лестница без него срезает нужное: минимизация НЕ
распространяется на `quality_floor` конфига (компоненты дизайн-системы, флаги
фич, тесты с регистрацией, проверки доступа через канонический стор). Экономия
строк — не цель и не метрика волны; цель — меньше возвратов на доработку.
Проверяет прохождение `cross-review` (пункт «переусложнение»), но ловить это на
ревью — уже дороже: там код написан и токены потрачены.

Для продуктовых скиллов (`feature-intake`, `product-flow`, `jira-task-standard`)
лестница НЕ применяется — там свои критерии краткости, про текст, а не про код.

**Режим тестов в волне:** в цикле разработки тикета — точечные прогоны по
затронутому (`pytest <путь>/-k`, `vitest related`); **полный сьют — один раз
на финале тикета** перед сдачей в Тестирование. Полный прогон на каждую
итерацию запрещён — жжёт время волны без прироста качества (финальный гейт
всё равно ловит). Красный полный прогон на финале → чинить, затем полный
прогон повторить.

## Будка (nyron-hub) — служебная шина мимо Jira

MCP-сервер плагина `nyron-hub` (тулзы `hub_*`, состояние — `.nyron-hub/` в
корне проекта, gitignored). Разделение каналов: **Jira — долгоживущее**
(брифы, отчёты, вердикты, статусы — аудит-лог), **будка — быстрое служебное**
(«взял/готово», бронь файлов, очередь мержа) — чтобы не засорять тикеты и не
тормозить об Jira. Волны: `hub_lock` на свои файлы при старте → `hub_unlock`
после пуша; `hub_merge_join` при готовности ветки. Диспетчер: `hub_status`
на каждом пробуждении (дешевле JQL), мерж — строго по голове очереди.
Механика — `references/wave-protocol.md`.

**Сдача = hub_post, полл — фолбэк.** Волна после пуша ветки/создания MR
ОБЯЗАНА запостить в будку `hub_post` с MR-ссылкой и тикетом — это основной
сигнал сдачи (SQLite-будка не теряет и хранит историю). Поллинг git/Jira у
диспетчера остаётся страховкой, не первым каналом: вотчеры умирают молча
(случай MR-полла НБ-1), будка — нет.

**Попутный фикс с поведением → мини-testgen той же изоляцией.** Найденный по
ходу тикета баг (ревью/адверсарка) чинится в волне (канон), но red-тест на
него пишет НЕ автор фикса — Sol или отдельный субагент, как на основной DoD.
Иначе «тесты под код» возвращаются через задний ход попутных багов (дыра
найдена на пилоте НБ-1 psylia).

**Будка одна на проект.** Якорь — каталог с `.claude/nyron-dev.md`, а не корень
git-репо: проект бывает зонтиком из независимых репо и worktree'ов, и по
git-корню сессии разъезжались по приватным будкам (диспетчер и волны переставали
видеть друг друга — баг прогона 21.07, чинено 0.4.1). Сомнение «мы в одной
будке?» проверяется одной командой в каждой сессии:
`${CLAUDE_PLUGIN_ROOT}/hub/hub-watch.sh hubdir` — пути обязаны совпадать.

## Соседние скиллы плагина (не дублировать, ссылаться)

- **`project-config`** — конфиг проекта; читать ПЕРВЫМ ходом.
- **`jira-task-standard`** — как писать и грумить тикеты; загружать при заведении
  и доведении задач.
- **`cross-review`** — обязательный шаг конвейера: GPT проверяет написанное
  Claude перед сдачей тикета в Ревью и на вердикте диспетчера.
- **`intake-groom`** — чистильщик входящих: сырьё → draft-тикет ДО груминга;
  draft-тикеты в волны не входят, пока человек не снял label `draft`.
- **`team-roles`** — ролевые карточки (продакт, UX, аналитик, техлид, SRE, QA)
  для продуктовой проработки и спец-задач волн.
- **`live-epic-testing`** — ручное live-тестирование эпика с пользователем
  (карта флоу, посекундная разметка). Это ДРУГОЙ режим: волны — массовое
  закрытие готовых тикетов; live-тестирование — диагностика и наполнение
  бэклога. Не смешивать; находки live-прогонов входят в волны уже тикетами.
- **`qa-handoff`** — передача закрытого эпика QA-инженеру: когда все тикеты
  эпика в «Готово» и predprod смоукнут, диспетчер собирает QA-тикет приёмки
  по этому скиллу. Эпик закрывается только после закрытия QA-тикета.
