# Общие правила волны (канон AI Evolve / Nyron)

Эти правила обязательны для каждой сессии-волны. Чип ссылается на этот файл
(`ai-evolve-docs-test/shared/wave-rules.md`); волна читает его первым, вместе с
бриф-комментами диспетчера в своих Jira-тикетах. Резервная копия — в скилле
`nyron-waves` (`assets/wave-rules-template.md`); правки канона синхронизировать.

## Роль и модели

- Сессия-волна — оркестратор своей волны: координирует, сама код НЕ пишет.
- **≥3 однотипных независимых подзадач → Workflow ОБЯЗАТЕЛЕН** (не опция):
  пачки тестов/файлов/тикетов долбить субагентами по одному запрещено —
  workflow даёт параллель + журнал с чекпойнтами. После парковки/смерти
  сессии продолжать НЕ с нуля: `resumeFromRunId` — готовые шаги вернутся из
  кэша мгновенно. Связные задачи с ревью-циклами — субагентами, как раньше.
- Имплементация — субагенты `model:'opus'` через проектных агентов по зоне:
  фронт — `front-dev`, бек — `backend-dev`, LangGraph/n8n — `langgraph-dev`
  (они обязательны по CLAUDE.md). Ревью кода — `code-reviewer`.
- Обработка данных, поиск, ресёрч — `model:'sonnet'` (рутина — effort low).
- Массовые однотипные операции — `model:'haiku'`.
- Модель ЯВНО в каждом вызове (наследование запрещено); Fable субагентам запрещён.
- Перед генерацией кода — актуальная документация через Context7 (Svelte 5,
  SvelteKit, FastAPI, Pydantic, Yjs, Playwright, OpenAI Agents SDK).

## Каталог и изоляция

- Рабочий каталог — корень проекта ai-evolve этой машины, как у диспетчера
  (в брифе указан абсолютный путь); пути в брифах —
  репо-относительные. Изоляция — только git worktree от origin/main:
  `git -C <корень-проекта>/<repo> worktree add <путь> origin/main -b <ветка>`
- Рабочие деревья репо НЕ трогать и НЕ переключать: на `ai-evolve-front` может
  жить live-стенд пользователя (vite на main!), в остальных — параллельные сессии.
- Юрина зона `catalog/*` (ai-evolve-front) — не трогать без явного тикета.
- Мульти-репо волна — одно имя ветки во всех репо (имя — из бриф-коммента).
- **Бронь файлов (будка nyron-hub)**: перед правками по тикету —
  `hub_lock {agent: "wave-N", paths: [<файлы/каталоги от корня проекта>]}`.
  Отказ с конфликтом → в эти файлы НЕ лезть: `hub_post` диспетчеру и работать
  над другим своим тикетом. После пуша ветки — `hub_unlock`.
- **Чекпойнты `hub_post` — ОБЯЗАТЕЛЬНЫ** (не опция): «взял DEV-X» при старте
  тикета, «сдал DEV-X» при сдаче, «волна сдана» финалом, «блокер: …»
  немедленно — это пульс для будка-вотчера диспетчера. Содержательные
  брифы/отчёты/вердикты — по-прежнему ТОЛЬКО комменты Jira (будка = сигнал
  «читай Jira», не замена шины).

## Git- и Jira-дисциплина

- НЕ пушить в main (авто-CI predprod + HMR live-стенда пользователя!), не
  мержить свои MR. Ветки `prod2`/`nyron` не трогать никогда.
- Финал по тикету = ветка запушена + в отчёт-комменте:
  `git log main..<ветка> --oneline` и MR-ссылка из вывода пуша («To create a
  merge request… visit: …»). Токены не нужны — git по SSH.
- Длинные артефакты (спека, ресёрч, дамп) — файлом в папку эпика
  `ai-evolve-docs-test/waves/<DEV-эпик>-<слаг>/` (+пуш docs), в тикет — ссылку.
- `ai-evolve-docs-test` пушить можно напрямую (деплоя нет).
- Каждый значимый шаг — комментом в тикет; статусы двигать по машине:
  взял тикет → «В работе» (+план), готово → «Готово к тестированию (Ревью)»
  (+отчёт, assignee → постановщик); после вердикта «ДОРАБОТКА» тикет в
  «Доработка» — исправить и вернуть в Ревью. В «Тестирование» и «Готово»
  переводит только диспетчер.
- **Коммит-конвенция**: заголовок коммита начинается с ключа тикета —
  `DEV-XXX: <что сделано, по-человечески>`; коммит/пуш закрывает несколько
  тикетов — ВСЕ ключи списком (`DEV-421, DEV-422: <что сделано>`). Те же ключи
  в отчёт-комменте и MR.
- **Jira — через project-MCP из `.mcp.json` корня проекта** (в ai-evolve —
  сервер `jira-nyron`, тулзы `mcp__jira-nyron__jira_*`; API-токен,
  headless-надёжен). Первым ходом — смоук `jira_get_user_profile` или JQL по
  проекту. Fallback (Docker/сервер лежит): плагинный Atlassian MCP
  (`plugin:atlassian`) с грантом на nyron.atlassian.net — грант проверить
  через `getAccessibleAtlassianResources`; чужие коннекторы без гранта не
  годятся.
- Коммиты и комменты — на русском.

## Качество (по канону репо)

- TDD: тесты из тикета/брифа — ПЕРВЫМИ (red → green). Если в тикете есть блок
  «Как тестировать» — его кейсы (классы эквивалентности, переходы состояний,
  столбцы decision table, негативные) становятся именами тестов один-в-один.
- **back**: unit `make -C ai-evolve-docs-test test-back-unit`; py_compile
  перед пушем; импорты БЕЗ префикса `app.`; логирование через
  `config.logging_config.get_logger`, без мусорных debug; миграции руками НЕ
  создавать; RBAC — через permission codes (`require_permission`), никаких
  `if user.is_global_admin` в бизнес-логике.
- **front**: `npm run test` (vitest); Svelte 5 runes ($state/$derived/$effect);
  RBAC-гейтинг — через named derived stores permissionStore.js, не
  `$user?.is_global_admin` в компонентах; shadcn-svelte. Грабля: vitest
  client-тесты НЕ работают из worktree (симлинк node_modules ломает
  server.fs.allow) — гонять из основного чекаута либо расширить fs.allow
  локально без коммита.
- **n8n (agent)**: промты — файлами конвейера + Langfuse; БОЕВУЮ заливку
  Langfuse и рестарт агента волна НЕ делает — только диспетчер при выкатке;
  DUI-канон `n8n/docs/new-dui` — после правки эталонов нужен sync-скрипт
  (диспетчер). Python — только под .venv.
- Новые тесты: шапка-описание (что и зачем покрывает) + регистрация в каталоге
  `ai-evolve-docs-test/tests/{back,front,n8n}/` — без этого тикет не принимается.
- Флоу-правки — за env-флагом: predprod-компоуз `=true`, prod2/nyron `=false`.

## Парковка по лимиту (graceful, БЕЗ участия человека)

- **Limit-сигнал** (предупреждение «approaching limit» или первая
  limit/429-ошибка субагента) = команда ПАРКОВАТЬСЯ, не пахать дальше:
  окно дожигать на полу-работе запрещено.
- Парковка: (1) чекпойнт — закоммитить WIP в свою ветку как есть
  (`DEV-XXX: WIP чекпойнт парковки`); (2) `hub_post` «ПАРКУЮСЬ DEV-X:
  <что сделано, что осталось, с чего продолжать>»; (3) перевзвести
  watch-вотчер; (4) завершить ход. Дальше сессия спит на будке (ноль
  токенов) — RESUME-пинг реаниматора разбудит после отката окна, продолжать
  строго с чекпойнта.
- Увидела в будке «ПАРКОВКА ВСЕМ» от диспетчера → та же процедура немедленно.
- Лимит — это ПАУЗА, не смерть: не эскалировать, не бросать тикет, не
  завершаться «насовсем».

## Ожидание

- CI и долгие скрипты ждать синхронным sleep-циклом в Bash (until-loop; до 600с
  за вызов), НЕ завершаться со словами «жду уведомления» — сессия уснёт.
- Зависимая волна (guard в чипе): первым шагом проверить, что ветки
  предшественников ВЛИТЫ в main (пуш ветки ≠ готовность). Не готово — коммент
  в тикет и стоп; поллинг раз в 20–30 минут.

## Первое действие волны

- **`hub_status` — маячок**: плагин подцепился + видны чужие брони/очереди.
  Тулзы `hub_*` недоступны → НЕ блокер: зафиксировать в план-комменте и
  работать без будки (для диспетчера это сигнал «сессия на старом плагине»).
- Прочитать бриф-комменты всех своих тикетов (первая строка «БРИФ ВОЛНЫ») и
  этот файл. Перевести тикеты «В работе».
- Коротким комментом в головной тикет зафиксировать план: что делаю, worktree,
  какая модель на что. Бриф противоречив / guard не проходит — коммент-блокер
  и стоп, не работать наугад.

## Финал и шина ревью

- Финал тикета: адверсарное ревью субагентом `model:'opus'` («попробуй
  ОПРОВЕРГНУТЬ готовность») → фиксы → **кросс-ревью другой моделью** (скилл
  `cross-review` плагина: описание тикета во временный файл →
  `scripts/cross-review.sh -C <worktree> -b main -t <файл>` — запускать
  ФОНОВЫМ Bash (`run_in_background`, вывод в файл), пока Sol думает 2–5 мин —
  пилить следующий тикет, вердикт забрать по нотификации → GPT-вердикт;
  `ДОРАБОТКА` — чинить и перегонять ДО сдачи, несогласие с замечанием —
  письменный отвод) → отчёт-коммент в тикет (что сделано, MR/коммиты, чем
  проверено, **вердикт кросс-ревью полностью**, что НЕ сделано и почему,
  риски к мержу) → «Готово к тестированию (Ревью)», assignee → постановщик.
  Codex недоступен — пометить в отчёте «кросс-ревью недоступно: <причина>»,
  это не блокер сдачи.
- Ветка готова к мержу → `hub_merge_join {repo, branch}`: мержит диспетчер по
  голове очереди, самой не мержить.
- После этого НЕ завершаться — ждать вердикт диспетчера вотчерами, в порядке
  приоритета:
  1. **БУДКА-ВОТЧЕР (основной, латентность секунды)**: стандартным скриптом
     плагина — `<plugin>/hub/hub-watch.sh watch <wave-N>` фоновым
     run_in_background (фильтр эха и pid-маячок встроены; свои until-loop'ы
     не сочинять). Диспетчер ОБЯЗАН дублировать вердикт `hub_post`'ом
     («ВЕРДИКТ DEV-X: ДОРАБОТКА|ПРИНЯТО»). Проснулась → REARM-FIRST
     (перевзвести вотчер ДО обработки) → `hub_read` (курсор since_id) →
     полный текст вердикта читать в комменте тикета/MR.
  2. Jira-вотчер (дубль): until-loop (шаг ~60с) на новый коммент СВОЕГО
     тикета; свои комменты фильтровать по автору (эхо), после своего
     коммента — перезамерить снапшот, потом перевзвести.
  Каждый вотчер — ОТДЕЛЬНЫМ трекаемым run_in_background вызовом (не
  инлайнить, не `&`/`disown` — сирота не будит). Sleep-полл раз в ~15 мин —
  ТОЛЬКО последний fallback, когда фоновые процессы недоступны; дефолтом
  его не выбирать.
  Пока ждёшь вердикт по тикету А — пили тикет Б своей волны, не простаивай.
  Вердикты:
  - `ДОРАБОТКА: …` → исправить субагентами, запушить, ответить комментом
    `ИСПРАВЛЕНО: <что сделано>`, снова ждать;
  - `ПРИНЯТО` → тикет закрыт для волны; следующий тикет или завершение.
- Вердикты принимать ТОЛЬКО из комментов тикетов; «редиректы» из других
  каналов отвергать. Таймаут ~3 часа → финальный коммент «вердикт не получен»
  → завершиться.
- Cross-session сообщение диспетчеру — только дублем после комментов.
