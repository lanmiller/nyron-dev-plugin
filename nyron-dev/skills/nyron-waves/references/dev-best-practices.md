# Бест-практисы разработки nyron-dev — перенос на любой проект

Канон «как мы разрабатываем», собранный из обкатки ai-evolve + внешнего
ресёрча (июль 2026). Это НЕ теория: каждый пункт либо чинил нашу реальную
боль, либо подтверждён независимыми данными. Настройка проекта — внизу.

## 1. Лестница до кода (из ponytail, но с ограничителем)

Три вопроса ДО первой строки, ответ в отчёт: нужен ли код вообще → нет ли уже
в базе (**пруф**: граф codebase-memory / grep, путь:строка) → нельзя ли
параметризацией существующего. Ограничитель обязателен: `quality_floor`
конфига не минимизируется, «меньше строк» — не аргумент и не метрика.
Почему не сам ponytail: независимая проверка (Scott Logic) показала, что
7 слов промпта дают тот же эффект, а «минимизируй всё» без стоп-ступеней
режет дизайн-систему и защитный код.

## 2. TDD-контур: тесты пишет ДРУГАЯ модель, код их не трогает

- Red-тесты из DoD пишет **Sol/codex** (чужая семья — слепые пятна не
  совпадают по построению; и не ест Claude-токены) в **репо тестов** — код
  Claude сдаёт экзамен по чужим билетам.
- Тесты = спека: чекпойнт сессии смотрит их ДО кода (покрывают DoD? не бред?).
- Код-агент тесты НЕ правит; несогласие с тестом = блокер наверх, не подгонка.
  Энфорс механический: дифф-валидатор в конвейере (дифф не касается тестовых
  путей). Kent Beck про агентов: «Genie не хочет TDD — он хочет код, а потом
  тесты, которые проходят»; агенты удаляют красный тест вместо починки кода.
- Обязательно для тикетов с поведением в DoD (баг → red-тест = репро);
  рефакторинг/инфра — арбитр существующий сьют (принцип «эталонного
  верификатора» из кейса C-компилятора Anthropic).

## 3. Режим тестовых прогонов

В цикле — точечно по затронутому или детерминированный сэмпл 1–10%
(`--fast`-паттерн Anthropic: сэмпл детерминирован для агента, случаен между
агентами — вместе покрывают всё). Полный сьют — ОДИН раз на финале тикета.
Полный прогон на каждую итерацию запрещён.

## 4. Модель-матрица и токены

Fable — решения (брифы, чекпойнты, вердикты, сложные конструкции, приёмка
пути пользователя). Opus — код и адверсарка. Sonnet — обработка/ресёрч.
Haiku — массовое и механические валидаторы. Sol — кросс-ревью и тест-ген.

- **Одна сессия — одна модель навсегда**: смена модели в сессии обнуляет
  KV-кэш (и человек забывает переключить). Разнообразие — через субагентов
  и Workflow-стадии.
- **Механика → Workflow, решения → сессия.** Дорогая модель в сессии
  окупается, только если механика не проходит через её контекст.
- Параллелизм — не бесплатный: ~15x токенов за мульти-агентность (данные
  Anthropic). Fan-out по потребности задачи, не «чем больше тем лучше».

## 5. Ревью-слои

- Кросс-модельное ревью (Claude пишет — GPT проверяет) — главный фильтр:
  «когда модель ревьюит свой код, она смотрит через ту же линзу, что его
  породила». Ревьюеру давать не только дифф: доступ к репо + карта flows
  (главный провал индустрии — diff-only ревью).
- Переусложнение (лишний слой, абстракция на вырост, подстилки под
  несуществующие случаи) — НЕБЛОКИРУЮЩЕЕ замечание; блокер из этой оперы
  только дубль механизма (DRY).
- НЕБЛОКИРУЮЩЕЕ не выбрасывается — автосбор в файл долгов проекта.
- Слои ревью добавлять по данным телеметрии, не по ощущению: избыточные
  ревьюеры дают повторы и режут пропускную способность, не качество.

## 6. Телеметрия вместо ощущений

Конвейер пишет `.nyron-hub/metrics.jsonl`: пруф лестницы (graph|grep|none),
находки по источникам (sol|adversarial|red-test|dispatcher), вердикты,
итерации. Решения о процессе — по таблице перекрытия за 2+ волны.
Метрики: возвраты на доработку, уникальные находки по источникам, время
бриф→ПРИНЯТО. LOC метрикой не является.

## 7. Настройка проекта (чек-лист переноса)

1. Плагин nyron-dev + `.claude/nyron-dev.md` (скилл `project-config`):
   `models.*`, `quality_floor`, `flows`, роли.
2. **Граф кода** (кодовые проекты): бинарь codebase-memory-mcp (GitHub
   DeusData, релиз под платформу, чексуммы) → запись в `.mcp.json` проекта
   (gitignored!) → `index_repository` каждого репо (секунды). Реиндекс при
   мерже — уже в каноне волн.
3. Репо/каталог тестов с каталогом-реестром (шапка-описание у каждого теста).
4. `.env.example` в каждом репо; секреты только из env, дефолтов-секретов
   в коде нет; целевое хранилище — GitLab CI/CD variables.
5. Политика кода приезжает сама SessionStart-хуком плагина — CLAUDE.md
   проекта её не дублирует.
