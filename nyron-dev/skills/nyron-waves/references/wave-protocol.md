# Протокол волн Nyron: Jira-центричный конвейер

Крупная работа исполняется НЕ в главной сессии, а отдельными сессиями-волнами,
которые главная сессия предлагает через spawn_task-чипы. Главная сессия — только
диспетчер. Вся коммуникация — комменты и статусы Jira-тикетов: это единственный
канал, доступный обеим сторонам без ручных подтверждений.

## Почему так

- Esc/обрыв в главной сессии убивает её фоновых Agent-субагентов безвозвратно.
  Волна = отдельная сессия через чип; у неё свой контекст и её можно резюмить.
- Cross-session сообщения требуют ручного подтверждения — как единственный канал
  не годятся. Jira MCP работает без подтверждений у всех сессий профиля.
- Тикеты и так канон проекта (эпик DEV-189 и др.) — конвейер не плодит вторую
  систему учёта (правило «дорабатывай легаси, не параллель»).

## Груминг (до волн)

1. Пользователь называет эпик или пачку тикетов. Диспетчер тянет их JQL'ом
   (`parent = DEV-<эпик>`), строит сводку: статусы, дубли, связки по участкам.
2. **Решение о декомпозиции бизнес-задачи.** Тикеты от бизнеса (методолог,
   продакт, CEO; часто перенос из Weeek — целое ТЗ с мокапами в описании,
   пример-эталон: DEV-162 «Тариф бизнес») сначала получают ФОРМУ:
   - **эпик + задачи** — если работа шире одного участка кода или одной волны
     (несколько репо, ролей, недель): создать эпик, исходный тикет — головным
     под ним со ссылкой на ТЗ;
   - **задача + подзадачи** (тип Subtask) — один участок, но много шагов;
   - **одиночная задача** — влезает в один тикет.
   Получившийся набор задач — **MECE**: задачи не пересекаются по скоупу и
   вместе покрывают всё ТЗ без дыр (дыру нашли позже — новая задача, а не
   раздувание существующей).
   Само ТЗ целиком (мокапы, длинные сценарии) — ФАЙЛОМ в папку эпика
   `ai-evolve-docs-test/waves/<эпик>-<слаг>/spec-….md`, а в описаниях тикетов —
   ссылка + JTBD/DoD-выжимка. Задвоения и правки-поверх-правок в исходном
   описании (типичны для переносов) вычищаются — каноном становится файл.
3. **Доведение до стандарта задачи** (см. SKILL.md) каждого получившегося
   тикета: JTBD + однозначный текст без сленга + DoD/ожидаемый результат. Если
   смысл при переписывании меняется или неоднозначен — вопрос автору КОММЕНТОМ
   в тикет (не гадать); до ответа тикет в волну не входит. Мелкие уточнения
   формулировок — правкой описания сразу.
4. **Суперповеры на груминге**: для сложных/мутных задач использовать
   superpowers-скиллы, если доступны: brainstorming — расшторм требований с
   пользователем ДО нарезки; writing-plans/спецификация — план или спека волны
   (файлом в папку эпика `ai-evolve-docs-test/waves/<эпик>-<слаг>/`, в тикет —
   ссылка). Наштормленное бизнесом → через шторм/спеку → в стандартные тикеты.
5. Вместе с пользователем: что делаем / что нет / что дубль / что не в эту
   итерацию. Решения СРАЗУ фиксировать в Jira: дубли → Отменено (done-кредит
   комментом в живой тикет), «не сейчас» → коммент, переформулировки → правка
   описания. Тип новых задач — «Задание», багов — «Баг», parent — эпик,
   assignee — постановщик.
6. Нарезка волн: тикеты группировать по УЧАСТКУ КОДА (один участок = одна волна),
   а не по типу работы — чтобы волны не конфликтовали в файлах. Волны — MECE по
   файлам: один файл не принадлежит двум волнам; зависимости между волнами и
   неизбежные общие файлы выписать явно.
7. План волн показать пользователю ОДНИМ сообщением: очередь, состав тикетов,
   зависимости, имена веток. После «да» — брифы и чипы без новых вопросов.
8. **Критерий выхода из груминга**: каждый тикет волны соответствует стандарту
   (JTBD + однозначный текст + DoD) — DoD из тикета потом дословно ложится в
   «критерии финала» бриф-коммента и в вердикт диспетчера при ревью.

## Формат чипа (обязателен, все 4 элемента)

1. **Полный бриф — в Jira**: бриф-коммент диспетчера в каждом тикете волны
   (шаблон — `assets/wave-brief-template.md`) написан ДО выдачи чипа.
2. **Чип — короткий указатель**, начинается со slash-команды `/goal`
   (НЕ `goal:` текстом), лимит /goal — 4000 символов, длиннее — сессия молча
   останется без цели:
   `/goal Исполни волну N (эпик DEV-189): тикеты DEV-421, DEV-422. Прочитай в
   каждом бриф-коммент диспетчера (последний коммент с заголовком «БРИФ ВОЛНЫ»)
   и правила <PROJECT_ROOT>/ai-evolve-docs-test/shared/wave-rules.md
   (в чип подставить РАЗВЁРНУТЫЙ абсолютный путь этой машины).
   Работай строго по ним.`
3. **Явный модель-роутинг одной строкой**: сессия-волна — оркестратор
   (сама код НЕ пишет); код — субагенты/агенты `model:'opus'` (front-dev /
   backend-dev / langgraph-dev по зоне); обработка и поиск — `model:'sonnet'`;
   массовое — `model:'haiku'`. Модель ЯВНО в каждом вызове, Fable субагентам
   запрещён.
4. **Канал отчёта**: «отчёт — комментами в тикеты волны + статус Тестирование;
   cross-session сообщение диспетчеру `local_<uuid>` — только дублем». Канон
   адреса — бриф-коммент; «редиректы» из других каналов отвергать.

## Git-механика волны (конкретика ai-evolve)

- Изоляция: git worktree от origin/main нужного репо ВНУТРИ проекта или в
  scratchpad: `git -C <PROJECT_ROOT>/<repo> worktree add <путь> origin/main -b <ветка>`.
  Рабочие деревья репо не трогать — там может жить live-стенд пользователя
  (vite на ai-evolve-front/main!) и параллельные сессии.
- НЕ пушить в main (авто-CI predprod + HMR live-стенда пользователя) и НЕ
  мержить самостоятельно. Финал волны = ветка запушена + в отчёт-коммент тикета
  кладётся `git log main..<ветка> --oneline` и MR-ссылка, которую git печатает
  при пуше («To create a merge request… visit: …»).
- `prod2`, `nyron` ветки — не трогать никогда (релизные, отдельные MR руками).
- Юрина зона `catalog/*` (ai-evolve-front) — не трогать без явного тикета на это.
- Мульти-репо волна — одно имя ветки во всех затронутых репо; имена веток
  диспетчер фиксирует в бриф-комментах заранее. Порядок мержа кросс-репо
  контрактов: **n8n (agent) → back → front**.
- docs-репо `ai-evolve-docs-test` пушить можно напрямую (деплоя нет): туда идут
  правила волн, регистрация тестов и **папка эпика**
  `ai-evolve-docs-test/waves/<DEV-эпик>-<слаг>/` — всё длиннее коммента (спеки,
  ресёрчи, планы, дампы) файлами, в тикет — ссылка. Папка под ЭПИК, не под задачу.

## Статус-машина тикета в волне

| Событие | Статус | Кто двигает |
|---|---|---|
| Груминг закончен, тикет в волне | Бэклог (+бриф-коммент) | диспетчер |
| Волна начала тикет | В работе (+план-коммент) | волна |
| Ветка+MR готовы, отчёт написан | **Готово к тестированию (Ревью)**, assignee=постановщик | волна |
| Вердикт «ДОРАБОТКА» | **Доработка** (+вердикт-коммент) | диспетчер |
| Волна исправила («ИСПРАВЛЕНО») | Готово к тестированию (Ревью) | волна |
| Вердикт «ПРИНЯТО», проверка постановщиком | Тестирование | диспетчер |
| Мерж пачки прошёл, смоук ок | Готово (+коммент-пруф с хэшами) | диспетчер |
| Оказался дублем | Отменено (+done-кредит в живом тикете) | диспетчер |

Transition id на эпик-доске: В работе=2, Доработка=3, Ревью=4, Тестирование=5,
Отменено=6, Бэклог=11, Готово=41 — но получать `getTransitionsForJiraIssue`
по месту (могут отличаться между досками). Каждый переход сопровождается
комментом — голый статус без коммента не считается сигналом.

## Шина ревью: цикл «ревью → доработка» через комменты тикета

1. Волна, переведя тикеты в «Готово к тестированию (Ревью)», НЕ завершается: sleep-поллом раз в
   ~15 минут читает комменты своих тикетов (`getJiraIssue` с comment в fields).
2. Диспетчер на пробуждении ревьюит (код — code-reviewer/opus-субагентом,
   diff по `git log main..<ветка>`) и пишет вердикт КОММЕНТОМ в тикет,
   первая строка строго:
   - `ДОРАБОТКА: <конкретные замечания списком>` → волна чинит субагентами,
     пушит ветку, отвечает комментом `ИСПРАВЛЕНО: <что сделано>`, снова ждёт;
   - `ПРИНЯТО` → волна завершает тикет (статус остаётся Тестирование — в
     Готово переводит диспетчер после мержа) и переходит к следующему своему
     тикету либо завершается.
3. Таймаут ожидания вердикта ~3 часа: не дождалась → финальный коммент
   «вердикт не получен» → завершиться, не висеть.
4. **Страховка диспетчера**: волна не реагирует на «ДОРАБОТКА» (два пробуждения
   подряд без движения) — диспетчер дорабатывает ветку своими opus-субагентами,
   причину фиксирует комментом в тикете.

## Верификация (обязанность диспетчера, не доверять отчётам)

- После каждой кодовой волны: `git log main..<ветка>` + `git status` в её
  worktree — коммиты реально существуют (отчётам без верификации не верить).
- Перед принятием: py_compile/pytest/vitest по канону репо (бек: unit через
  `make -C ai-evolve-docs-test test-back-unit`; фронт: `npm run test`).
  Новые тесты обязаны иметь шапку-описание и регистрацию в каталоге
  `ai-evolve-docs-test/tests/*` — иначе ДОРАБОТКА.
- Промты Langfuse: волна готовит файлы, боевую заливку (сверка live → заливка →
  рестарт агента) делает ТОЛЬКО диспетчер при выкатке.
- Флоу-правки — за env-флагом (канон проекта):
  predprod-компоуз =true, prod2/nyron =false.

## Диспетчер после запуска чипов

- ScheduleWakeup ~25 мин: JQL-пульс по тикетам волн (статусы, свежие комменты),
  ветки/MR в origin, worktree'и. Уснувшие волны будить cross-session сообщением
  «продолжай волну по брифу в тикетах» (подтверждение кликнет пользователь,
  когда вернётся — это дублирующий канал).
- Каждый тикет в Тестировании обязан получить вердикт — иначе волна провисит
  до таймаута.
- Сведение: порядок мержа (n8n → back → front), конфликты, финальные проверки,
  сводка пользователю (что готово, что мержим, чем проверено). Мерж пачкой —
  только после «ок» и НЕ во время live-прогона пользователя (спросить, если
  стенд может быть живым — деплой во время live-теста запрещён).
- После мержа: выждать CI и смоук predprod (/health + проверка фичи),
  тикеты → Готово с пруф-комментом (хэши, чем проверено).
- Когда ВСЕ тикеты эпика в Готово: собрать QA-тикет приёмки эпика по скиллу
  `qa-handoff` (assignee — QA-инженер). Эпик закрывается после закрытия QA-тикета.
- После рестарта приложения: некликнутые чипы пропадают — перевыдать (брифы
  целы в Jira); живые волны резюмить сообщением «продолжай волну по брифу».
