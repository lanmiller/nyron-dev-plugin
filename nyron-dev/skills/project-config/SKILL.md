# project-config — один плагин, много проектов

Всё «как делать работу» живёт в скиллах плагина и одинаково для всех проектов.
Всё «чем проекты отличаются» живёт в конфиг-файле проекта. Этот скилл — про
чтение и создание конфига.

## Два файла конфига

| Файл | Кто | Git |
|---|---|---|
| `.claude/nyron-dev.md` | канон проекта, общий для команды, БЕЗ секретов | коммитится |
| `.claude/nyron-dev.local.md` | личные оверрайды (тот же формат) | gitignored |

Формат: YAML-фронтматтер (структура) + свободный markdown (заметки проекта для
агентов). Схема полей — `assets/config-template.md`. Значения `.local.md`
накладываются ПОВЕРХ канона по ключам верхнего уровня.

## Чтение конфига (протокол для всех скиллов плагина)

1. Прочитать `.claude/nyron-dev.md` от корня проекта; поверх — `.local.md`,
   если есть.
2. Файла нет → работать по дефолтам ai-evolve (текущие якоря скиллов) И
   предложить пользователю создать конфиг этим скиллом.
3. Ключевые развилки поведения:
   - `storage: jira` → задачи/брифы/отчёты в Jira через `jira.mcp_server`;
     `storage: files` → задачи markdown-файлами в `files.tasks_dir`
     (фронтматтер: status, assignee, ticket-слаг; статусы двигаются правкой
     фронтматтера, отчёты — секциями файла).
   - `type: development` → волны пишут код (git-механика, ветки, MR);
     `type: research` → волны работают с данными: гипотеза → эксперимент →
     замер; артефакты — файлы в `docs_dir`, git — по ситуации.
   - `models.*` → подставлять в чипы и вызовы субагентов вместо зашитых
     дефолтов.
   - `reviewer.engine` → включает/выключает шаг кросс-ревью (скилл
     `cross-review`).
   - `roles.<slug>` → оверрайды карточек скилла `team-roles`.

## Создание конфига под новый проект

1. Скопировать структуру из `assets/config-template.md`.
2. Спросить у пользователя минимум (не растягивать интервью): тип проекта,
   хранилище задач (есть ли Jira и какой project key), чем занимаются рабочие
   (код/данные), какие роли включить.
3. Заполнить `roles.*.context` реальными путями проекта («куда смотреть»
   каждой роли: папки доков, дашборды, канон-файлы) — это главная проектная
   ценность конфига; пустые context-списки не оставлять, лучше спросить.
4. Записать `.claude/nyron-dev.md`, добавить `.claude/nyron-dev.local.md` в
   `.gitignore` проекта, туда же `.nyron-hub/` (состояние будки).
5. Показать пользователю итоговый конфиг одним сообщением.

## Jira MCP проекта (для storage: jira)

Канон доступа в Jira для ВСЕХ проектов — **локальный project-MCP в
`.mcp.json` корня проекта** (докерный `mcp-atlassian` по API-токену:
работает в headless-сессиях, без OAuth-капризов). Плагинный/облачный
Atlassian-коннектор — только интерактивная альтернатива.

При создании конфига проекта со `storage: jira`:
1. Проверить `.mcp.json` корня проекта: есть ли сервер с именем из
   `jira.mcp_server`.
2. Нет — создать запись из `assets/jira-mcp-template.json`: подставить site
   и email, токен взять из `~/psylia-secrets` (спросить пользователя, какой
   файл). Требуется запущенный Docker.
3. Убедиться, что `.mcp.json` в `.gitignore` проекта (там токен!).
4. Смоук: вызвать `jira_get_user_profile` — сервер отвечает.

## Каноны проекта для codex (AGENTS.md)

Codex (cross-review, qa-run, codex-тул) НЕ читает CLAUDE.md — его конвенция
`AGENTS.md` (та же иерархия: `~/.codex/AGENTS.md` → git-root → подпапки,
лимит ~32KiB на файл). При настройке проекта: в корне и в каждом код-репо,
где есть `CLAUDE.md`, создать локальный симлинк `ln -s CLAUDE.md AGENTS.md`
(не коммитить, если пуш триггерит CI — симлинк восстанавливается этим скиллом
на каждой машине). Тогда каноны проекта автоматически уходят в GPT при любом
запуске codex в этом каталоге.

## Проектные MCP для codex (.codex/config.toml)

У codex есть нативный проектный конфиг: `<проект>/.codex/config.toml`
(читается ТОЛЬКО для trusted-проектов). Через него проекту отдаются его
MCP-серверы (обсидиан-волт, smart-поиск и т.п.) — у каждого проекта свои
порты/ключи/волты. Настройка (рецепт обкатан 18.07.2026, codex 0.131):

1. Trust проекта — в `~/.codex/config.toml`:
   `[projects."<абсолютный-путь>"]` → `trust_level = "trusted"`.
2. `<проект>/.codex/config.toml` — секции `[mcp_servers.<имя>]`
   (command/args/env) + **ОБЯЗАТЕЛЬНО** `default_tools_approval_mode =
   "approve"` в каждой (значение именно `approve`; `auto` молча игнорируется,
   и в headless `codex exec` вызовы умирают с «user cancelled MCP tool call»).
3. **Грабля болтливых серверов**: MCP-сервер, пишущий служебный текст в
   stdout вперемешку с JSON-RPC (например smart-connections), ломает строгий
   парсер codex. Лечение — bash-обёртка `... | grep --line-buffered '^{'`
   (пример: `.codex/smart-connections-wrapper.sh` в ai-evolve) +
   `startup_timeout_sec = 60`, если сервер долго грузится.
4. `.codex/` — в `.gitignore` проекта (внутри ключи).
5. Смоук: `codex mcp list` из каталога проекта показывает серверы; живой
   вызов — `echo "вызови <тул> ..." | codex exec --sandbox read-only -`.

## Секреты

В конфиг секреты НЕ пишутся никогда. Токены (Jira API и пр.) живут в
`~/psylia-secrets` или локальном окружении машины; конфиг может лишь называть
ИМЯ MCP-сервера, который ими пользуется.

## Смежные скиллы

- `team-roles` — читает `roles.*` оверрайды отсюда.
- `cross-review` — читает `reviewer.*`.
- `nyron-waves` — читает `storage`, `models`, `type` при нарезке волн.
- `intake-groom` — читает `storage` (куда класть черновик задачи).
