# Стандарт Jira-задачи AI Evolve / Nyron

Скилл — единый канон «как писать таски» для людей и агентов. Им пользуются:
человек, заводящий тикет; диспетчер волн на груминге (`nyron-waves`);
live-тестирование при оформлении находок (`live-epic-testing`); сборка
QA-приёмки (`qa-handoff`).

## Конвенции трекера — из конфига проекта

Имена типов, лейблов и статусов у каждого продукта свои. Брать их из конфига
(скилл `project-config`): `jira.site`, `jira.project_key`, `jira.issue_types`,
`jira.labels`, `jira.statuses`, `jira.routes`. **Поля нет → дефолты ai-evolve**
(таблица ниже) — старые конфиги продолжают работать без правок.

| Что | Ключ конфига | Дефолт (ai-evolve) |
|---|---|---|
| Трекер и проект | `jira.site` / `jira.project_key` | nyron.atlassian.net, DEV |
| Типы | `jira.issue_types` | «Задание» (НЕ «Задача»), «Баг», «Subtask», «Эпик» |
| Лейблы зоны | `jira.labels` | `front` / `back` / `agent` / `infra` / `catalog` |
| Маршрут отдела | `jira.routes` | один проект = `jira.project_key` |
| Папка эпика | `epic_folder` | `ai-evolve-docs-test/waves/{key}-{slug}` |

Общее для всех проектов:

- `qa`-лейбл — только для явных задач QA-инженеру.
- Эпики — через parent. Assignee = постановщик (кто принимает результат).
- Дубли не плодить: перед созданием сверить с эпиком (JQL `parent = <эпик>`).
  Статус «Отменено» — только у дублей, с done-кредитом комментом в живом тикете.
- Имена статусов различаются между досками («Бэклог» vs «К выполнению») —
  ориентироваться на категорию (new / indeterminate / done), transition id
  получать `getTransitionsForJiraIssue` по месту.
- `storage: files` → тикета нет: задача = файл в `files.tasks_dir` с тем же
  содержимым, статусы двигаются правкой фронтматтера.

## Обязательная структура тикета

Шаблон с полями и примером «плохо → хорошо» — `assets/task-template.md`.
Кратко:

1. **Заголовок** — глагольная фраза результата (пирамида: вывод первым).
2. **JTBD** — «когда <ситуация>, я хочу <действие>, чтобы <ценность>».
3. **Контекст** — что известно, с пруфами; без сленга.
4. **Что сделать** — однозначная постановка.
5. **DoD / ожидаемый результат** — MECE-список наблюдаемых поведений с пруфами.
6. **Как тестировать** — обязателен для багов и фич с поведением; техника
   выбирается по `references/test-design.md` (Equivalence Partitioning /
   State Transition / Decision Table / сценарий). Кейсы блока трижды
   переиспользуются: имена dev-тестов → чек-лист QA-приёмки → регресс.
7. **Вне скоупа** — что сознательно не делаем (опционально).
8. **Техконтекст** — технический слой для разработчиков/агентов (пути, логи,
   гипотезы о корне); единственное место, где допустим сленг.

## Стиль (McKinsey)

- **Пирамида**: вывод/результат первым — заголовок и первая строка отвечают
  «что получим».
- **Кратко и однозначно**: одно предложение — одна мысль; двоякое чтение —
  переписать; без воды и «и т.д.».
- **Без сленга в человеческом слое**: текст читается методологом, продактом,
  CEO с первого раза. Сугубо техническая задача (рефакторинг, инфра) всё равно
  получает человеческий верх в 1-2 строки: зачем это бизнесу/команде.
- **MECE**: пункты DoD не пересекаются и покрывают всю готовность; задачи
  внутри эпика не пересекаются по скоупу и покрывают всё ТЗ без дыр (нашли
  дыру — новая задача, а не раздувание существующей).

## Приём бизнес-задачи (наштормленной)

> Пришла **решённая фича / бизнес-требование** («надо добавить X»), а не готовое
> ТЗ на тикет → сначала скилл **`feature-intake`**: требование как артефакт
> (файл) → согласование → эпик. Сюда возвращаемся, когда пора оформлять тикеты.

Задачи от методолога, продакта, CEO (часто перенос из Weeek: целое ТЗ с
мокапами и правками-поверх-правок в описании) — НЕ отбрасывать, а доводить:

1. **Решить форму** (декомпозиция):
   - **эпик + задачи** — работа шире одного участка кода/одной итерации
     (несколько репо, ролей, недель);
   - **задача + подзадачи** — один участок, много шагов;
   - **одиночная задача** — влезает в тикет.
2. **ТЗ целиком** (мокапы, сценарии) — файлом в папку эпика
   (`epic_folder` конфига; дефолт
   `ai-evolve-docs-test/waves/<DEV-эпик>-<слаг>/spec-….md`), в тикетах — ссылка
   и JTBD/DoD-выжимка. Задвоения в исходном описании вычищаются, канон = файл.
3. **Довести каждый тикет** до структуры выше. Смысл меняется или неоднозначен —
   вопрос автору КОММЕНТОМ в тикет, не гадать; до ответа тикет в работу не берётся.

## Суперповеры в флоу (если установлен плагин superpowers)

Встраивать по ситуации, с мягкой деградацией (нет плагина — пропустить шаг):

- Требования мутные, «хочется вот такого» → **superpowers:brainstorming**
  ДО декомпозиции: расшторм с постановщиком, итог — согласованные требования.
- Большая фича, нужна спека/план → **superpowers:writing-plans**: спека файлом
  в папку эпика, в тикеты — ссылка.
- Готовая спека в Confluence → скилл **atlassian:spec-to-backlog** как черновик
  декомпозиции, затем довести тикеты до этого стандарта (он создаёт сырые).

## Чек перед сохранением тикета

- [ ] заголовок = результат; JTBD есть; DoD — MECE и проверяем;
- [ ] человеческий слой без сленга; технарское — в Техконтексте;
- [ ] «Как тестировать» есть (для багов/фич с поведением);
- [ ] parent/лейблы/assignee выставлены; дубль проверен по эпику;
- [ ] длинное — файлом в папку эпика, не в описании.
