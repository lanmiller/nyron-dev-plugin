## Принципы письма (McKinsey + MECE)

- **Пирамида**: вывод первым. Заголовок = глагольная фраза результата
  («Вернуть кнопку „Перезапустить" на карточку»), не «Проблема с кнопкой»;
  первая строка описания отвечает «что получим».
- **Кратко и однозначно**: без воды, вводных слов и «и т.д.»; одно предложение —
  одна мысль; если фразу можно понять двояко — переписать.
- **Без сленга** в человеческом слое: текст читается методологом, продактом,
  CEO с первого раза. Кодовые слова, пути, логи — только в блоке «Техконтекст».
- **MECE**: пункты DoD не пересекаются и вместе покрывают всю готовность;
  задачи внутри эпика не пересекаются по скоупу и покрывают всё ТЗ.
- **Два слоя**: верх (JTBD → Что сделать → DoD) — для любого члена команды;
  низ («Техконтекст») — для разработчиков/агентов, там допустима полная
  техническая точность. Сугубо техническая задача (рефакторинг, инфра) всё
  равно получает человеческий верх в одну-две строки: зачем это бизнесу/команде.

**JTBD:** Когда <ситуация/триггер>, я хочу <действие/возможность>,
чтобы <ценность/зачем>.

**Контекст:** <что известно: где воспроизводится, пруфы (лог/скрин/линк на
файл в папке эпика), связанные тикеты. Без внутреннего сленга — текст должен
читаться методологом/продактом/CEO без словаря.>

**Что сделать:** <однозначная постановка; для бага — «симптом → ожидание →
куда смотреть», для задачи — рамки решения без навязывания реализации.>

**DoD / ожидаемый результат:**
- <наблюдаемое поведение 1 — как проверяем, что готово>
- <наблюдаемое поведение 2 / пруф: лог-маркер, тест, скрин>

**Как тестировать** (обязателен для багов и фич с поведением; техника по
правилу из references/test-design.md — EP / State Transition / Decision Table /
сценарий):
- <кейс 1>
- <кейс 2>
- негативные: <что должно отбиваться>

**Вне скоупа:** <что сознательно НЕ делаем в этом тикете (опционально).>

**Техконтекст** (для разработчиков/агентов; опционально):
<пути файлов, лог-маркеры, гипотезы о корне, ссылки на код — здесь можно
точно и по-технарски.>

## Пример «плохо → хорошо»

**Плохо** (не пройдёт груминг):
> Кнопка рестарта отвалилась опять после плей реди фиксов, надо чинить флаг
> который бек не снимает на хедлес-пути, ну ты понял

**Хорошо:**
> **Заголовок:** Вернуть кнопку «Перезапустить» на карточку плейбука после прохождения
>
> **JTBD:** Когда я прошёл плейбук до конца, я хочу видеть на карточке кнопку
> «Перезапустить», чтобы пройти его заново с новыми вводными.
>
> **Контекст:** После входа в режим прохождения новым (автоматическим) путём
> кнопка исчезает навсегда: признак «первая сборка» на карточке снимается
> только старым путём входа, новый его не снимает.
>
> **Что сделать:** Снимать признак «первая сборка» при автоматическом входе
> в прохождение.
>
> **DoD (MECE):**
> - после полного прохождения плейбука владелец видит кнопку «Перезапустить»;
> - признак «первая сборка» снят после первого входа в прохождение
>   (проверка по данным карточки);
> - на снятие признака есть юнит-тест, зарегистрированный в каталоге тестов.
>
> **Техконтекст:** флаг `content.is_first_build` (карточка, Mongo);
> снимался в `_apply_play_restart` (reset-путь), новый вход — headless
> `play_ready_intro` (бек, 8a7fff20) его не проходит; пруф: доска 6a58ba86,
> `is_first_build=true` при `stage=ready`; гейт кнопки —
> PlaybookCardNode.svelte:694 `{#if !firstBuild}`.
