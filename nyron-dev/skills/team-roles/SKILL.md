# team-roles — роли как скиллы

Скилл — это инструкция «как делать вот эту работу». Роль — тонкая карточка:
миссия, когда зовут, вход/выход, чек-лист. Карточки в плагине **generic**;
всё проектное — из конфига (`roles.<slug>` в `.claude/nyron-dev.md`).

## Протокол надевания роли

1. Прочитать конфиг проекта (скилл `project-config`), секцию `roles.<slug>`:
   - `enabled: false` → роль в этом проекте не используется, сказать об этом;
   - `model` → если роль исполняет субагент/чип — модель брать отсюда;
   - `context` → **прочитать перечисленные пути ПЕРЕД работой** — это «куда
     смотреть» в данном проекте (доки, каноны, дашборды);
   - `notes` → проектные особенности роли, учитывать как аксиомы.
2. Прочитать карточку роли из `references/<slug>.md`.
3. Работать строго в границах карточки: роль делает свой ВЫХОД и не лезет в
   чужой. Нужен чужой выход — сказать, кого позвать, а не подменять.

## Состав команды

| Роль | slug | Борт | Модель (дефолт) |
|---|---|---|---|
| Продакт | `product` | продуктовый (шаг 1) | fable (точечно) |
| UX-ресёрчер | `ux-researcher` | продуктовый | opus |
| UX-дизайнер | `ux-designer` | продуктовый | opus |
| Маркетолог / CRM | `marketer` | продуктовый | opus |
| Рыночный ресёрчер | `market-researcher` | продуктовый | opus (`method: deep-research`) |
| Дата-аналитик | `data-analyst` | продуктовый / research | sonnet |
| Техлид | `tech-lead` | доставка (шаг 2) | opus |
| SRE-лид | `sre-lead` | доставка | opus |
| QA | `qa` | доставка | sonnet |

Два борта = два шага системы: **продуктовый** решает ЧТО делать и выдаёт
понятную задачу; **доставка** исполняет. Дорогая модель (fable) — только на
реально сложное продумывание и ненадолго; рутина — дешёвыми.

Роли-исключения, которые НЕ здесь: **диспетчер** — это режим сессии
(`nyron-waves`), **ревьюер** — другая модель (`cross-review`), **чистильщик**
— конвейер (`intake-groom`).

## Продуктовый цикл (шаг 1, левая половина системы)

Типовой маршрут сырой фичи до задачи в бэклоге:

```
сырьё → intake-groom (черновик) → product (JTBD, ценность, скоуп)
      → [ux-designer | data-analyst — по потребности]
      → jira-task-standard (постановка) → бэклог → nyron-waves (шаг 2)
```

Человек участвует в точках решений (что делаем, что готово); механику тянут
роли. Чем чётче выход продуктового борта — тем меньше человека дёргает
доставка.

## Карточки

- **`references/product.md`** — продакт: ценность, приоритет, скоуп.
- **`references/ux-researcher.md`** — поведение и джобы пользователя (JTBD),
  дешёвый тест; НЕ экраны.
- **`references/ux-designer.md`** — UX: сценарии, флоу, состояния экранов.
- **`references/marketer.md`** — коммуникации, офферы, рассылки, сегменты.
- **`references/market-researcher.md`** — внешний мир: конкуренты, отзывы,
  статьи, рыночный контекст; метод — скилл `deep-research`.
- **`references/data-analyst.md`** — аналитик: данные, гипотезы, замеры.
- **`references/tech-lead.md`** — техлид: архитектура решения, риски, нарезка.
- **`references/sre-lead.md`** — SRE: деплой, мониторинг, откат, инциденты.
- **`references/qa.md`** — QA: тест-дизайн, приёмка, регресс.

- **`references/methodologies.md`** — общий метод для всех ролей: HADI, JTBD,
  Lean, Data-Driven, McKinsey-пирамида + MECE (что это · как применять ·
  чек-вопросы роли). Роль работает по методу, а не «сделай хорошо».

Карточки — скелет, который дополняется из практики: роль плохо отработала →
уточнить карточку (generic-часть) или конфиг проекта (проектная часть) — тем
же заходом, не откладывая.
