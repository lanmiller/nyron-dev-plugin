# Методологии продукта

> Как продуктовые роли работают — по названным методам, не «от балды». Читают
> все продуктовые роли (подключается через `roles.*.context` в конфиге
> проекта). По каждой методологии: **что это · как применяем · чек-вопросы
> роли.** Скилл `product-flow` формулирует продуктовый флоу как HADI-петлю.

---

## HADI — костяк флоу (петля, не труба)

**Что это.** Цикл проверки гипотез: **H**ypothesis → **A**ction → **D**ata →
**I**nsight. Инсайт закрывает петлю — питает следующую гипотезу.

**Как применяем.** Стадии скилла `product-flow` (`черновик → проработка →
челлендж → отдана → раскатана → замер → вердикт`) — это один оборот HADI:

| HADI | Стадии product-flow |
|---|---|
| **H** — что верим и почему (с цифрой) | черновик, проработка, челлендж |
| **A** — что делаем | отдана, раскатана |
| **D** — что меряем (**заявлено заранее**) | замер |
| **I** — что узнали → решение | вердикт |

**Правило петли.** Вердикт всегда даёт **инсайт**, не только «зашло/не
зашло»: что именно узнали про пользователя или продукт. Инсайт идёт в канон
проекта (`context`/журнал из конфига) и становится основанием следующей
гипотезы.

**Чек-вопросы:**
- [ ] Гипотеза сформулирована как проверяемое утверждение, а не как задача?
- [ ] Метрика **D** названа ДО раскатки (не подбираем постфактум)?
- [ ] Вердикт даёт инсайт, который можно переиспользовать?

## JTBD — работа пользователя, не фича

**Что это.** Jobs to be Done: пользователь «нанимает» продукт сделать работу.
Формулируем от работы, а не от фичи. «Когда `<ситуация>`, я хочу
`<действие>`, чтобы `<результат>`».

**Как применяем.** Ведут product + ux-researcher. Пример (пример, не из
проекта): не «добавить кнопку экспорта», а «когда я закончил отчётный
период, хочу выгрузить данные одним кликом, чтобы не собирать их вручную».

**Чек-вопросы:**
- [ ] Формулировка про пользователя и его ситуацию, а не про нашу фичу?
- [ ] Понятна роли из другого борта без объяснений?
- [ ] Есть реальная «работа», или мы придумали спрос?

## Lean — дешёвый тест первым (Build-Measure-Learn)

**Что это.** Build-Measure-Learn: минимальный эксперимент, который даёт ответ
дешевле всего. Не строим полную механику, чтобы проверить идею.

**Как применяем.** Правило: **до полной раскатки — минимальная проверка**
(узкий сегмент / A/B / заглушка / ручной прогон). Ведут product +
ux-researcher. Пример (пример): заглушку будущей фичи показали до
разработки — замерили спрос по клику прежде чем строить.

**Чек-вопросы:**
- [ ] Какой самый дешёвый способ узнать, что гипотеза верна?
- [ ] Можно проверить на узком сегменте / A/B, а не раскатывать на всех?
- [ ] Что именно этот тест докажет или опровергнет?

---

## Data-Driven — жёсткий гейт на цифрах

**Что это.** Решения из данных, а не из мнений. Каждый вывод — с цифрой и
источником.

**Как применяем.** Гипотеза **не проходит проработку** без двух чисел:
**база** (сколько сейчас) и **размер эффекта** (сколько на кону — деньги,
пользователи, время). Ведёт data-analyst. Где данные и как их достать — в
`context` роли из конфига проекта. Пример из практики (пример): «нет
удержания в сегменте» оказался одной точкой воронки, а не продуктом —
потому что посчитали, а не поверили.

**Чек-вопросы:**
- [ ] Есть база (текущее значение) и размер эффекта в конкретных числах?
- [ ] Каждый вывод подкреплён цифрой с источником и датой?
- [ ] Проверено против прежних выводов проекта (не спорим о закрытом)?

## McKinsey (пирамида) + MECE — пишем для людей

**Что это.** Пирамида: вывод/ответ первым, потом обоснование. MECE: пункты не
пересекаются (Mutually Exclusive) и вместе покрывают всё (Collectively
Exhaustive).

**Как применяем.** Обязательно — карточки и задачи читают разные роли и
отделы, должны понять с первого раза. Ведут все. Стиль, словарь терминов и
прежние решения проекта — там, где указано в `context` конфига.
- **Пирамида:** первая строка блока = ответ. Заголовок = результат, не тема.
- **MECE:** блоки (например «разработка/дизайн/коммуникации») не
  пересекаются и покрывают весь скоуп; нашли дыру — новый пункт, а не
  раздувание существующего.
- **Язык:** отраслевые термины проекта — ок; канцелярит и выпендрёж — нет.

**Чек-вопросы:**
- [ ] Первая строка отвечает «что получим», а не вводит в тему?
- [ ] Пункты MECE — без пересечений и без дыр?
- [ ] Смежная роль поймёт без переспроса?

## Раскладка по ролям

| Роль | Ведущие методологии |
|---|---|
| Продакт | HADI (владелец цикла) · JTBD · Lean · McKinsey+MECE |
| Дата-аналитик | Data-Driven · HADI (шаги D+I) |
| UX-ресёрчер | JTBD · Lean (дешёвый тест) |
| Рыночный ресёрчер | Data-Driven (источник+дата) · Lean (дешёвая разведка) |
| Маркетолог | Data-Driven (сегменты/офферы) · McKinsey (тексты кратко) |
| Техлид | MECE-декомпозиция |

## Смежное

- Флоу гипотезы (HADI-петля, 7 стадий) — скилл `product-flow`.
- Роли, которые применяют методологии, — скилл `team-roles`.
- Стиль, словарь и прежние решения проекта — `context` в конфиге
  (`project-config`).
