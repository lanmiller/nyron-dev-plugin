# QA-приёмка эпика (передача QA-инженеру)

> **Модель сессии — дорогая (`models.thinker`, Fable).** Приёмка — это путь
> пользователя целиком, не построчная проверка: здесь нужна «конструкция в
> целом» — где флоу рвётся, что забыли, какой сценарий не покрыт. Дешёвой
> моделью собирается формальный чек-лист, дорогой — карта реальных дыр.

Контекст команды: dev-цикл (мы) сам себе QA — TDD, ревью, live-прогоны,
смоук на predprod. QA-инженер (Агиль Гасанов, QA-отдел) включается ПОСЛЕ:
перепроверка на predprod свежим взглядом + его зона — автотесты и
тест-сценарии всего продукта. Скилл собирает для него ОДИН самодостаточный
QA-тикет на эпик — не 30 техтикетов с жаргоном.

## Когда собирать

Все условия сразу:
- все тикеты эпика в «Готово» (или явно вынесены из скоупа комментом);
- код смержен, CI прошёл, predprod смоукнут командой.

**Сигналов передачи два (гибрид, оба обязательны):**
1. **сам ЭПИК → статус «Готово к тестированию (Ревью)»** (реальное имя статуса
   в воркфлоу DEV) — сигнал на доске, виден без открывания;
2. **один QA-тикет на эпик** (ниже) — рабочий юнит QA-инженера: его очередь по
   label `qa`, трекаемый прогресс чек-листа, статистика QA-отдела.

Анти-паттерн: тестировочный тикет на КАЖДЫЙ дев-тикет — засоряет доску вдвое,
не делать никогда. Один QA-тикет на эпик — норма.

## Сборка QA-тикета

Тип «Задание», parent = эпик, **label `qa`**, assignee = QA-инженер.
Шаблон — `assets/qa-ticket-template.md`. Содержимое агрегируется из закрытых
тикетов эпика ТОЛЬКО человеческим слоем (Техконтексты не тащить):

1. **Что появилось/изменилось** — по фичам, языком пользователя (из JTBD
   закрытых тикетов).
2. **Окружение** — стенд (predprod URL), тестовый аккаунт/доступы, с чего
   начать (конкретная доска/раздел).
3. **Чек-лист проверки** — склейка блоков «Как тестировать» всех тикетов
   эпика, сгруппированная ПО СЦЕНАРИЯМ ПОЛЬЗОВАТЕЛЯ (не по тикетам):
   happy-path'ы, границы, негативные. У тикета нет блока «Как тестировать» —
   ДОПИСАТЬ его при сборке (по jira-task-standard), а не пропустить.
4. **Известные ограничения** — из «Вне скоупа» тикетов + сознательные хвосты.
5. **Материалы для автотестов** — та же таблица кейсов в машинном виде
   (сценарий → шаги → ожидание): основа тест-сценариев и автотестов продукта,
   которые ведёт QA. Если по эпику есть карта флоу live-прогонов — ссылка:
   это готовая трасса переходов для тест-дизайна.
6. **Куда заводить находки** — новые баги отдельными тикетами в эпик по
   стандарту `jira-task-standard` (QA-инженеру достаточно человеческого слоя:
   симптом → шаги → ожидание; Техконтекст допишет команда на триаже).

## Правила после передачи

- Находки QA **не реоткрывают** закрытые тикеты — только новые баги в эпик;
  триаж — на ближайшем груминге (диспетчер/постановщик).
- QA-тикет закрывает сам QA-инженер, когда чек-лист пройден.
- Эпик закрывается ТОЛЬКО после закрытия QA-тикета (статус эпика:
  «Готово к тестированию (Ревью)» → «Готово»).
- Вопросы QA ↔ команда — комментами в QA-тикете (единая шина, как везде
  в nyron-dev).

## Чек перед отправкой QA-тикета

- [ ] читается без словаря: ни одного внутреннего термина без расшифровки;
- [ ] чек-лист покрывает ВСЕ фичи эпика (MECE по сценариям);
- [ ] окружение и доступы проверены (аккаунт живой, стенд на нужной версии);
- [ ] ограничения честно перечислены (иначе QA заведёт «баги» на известное);
- [ ] assignee = QA-инженер, label qa, parent = эпик.
