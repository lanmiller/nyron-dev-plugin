# product-flow — гипотеза как петля HADI

Половина продуктовой работы — не «решено сделать», а «верим, что X улучшит Y».
У такой работы есть **основание, замер и вердикт**, и она **закольцована**:
вывод одной гипотезы — вход следующей. Единица трека — **гипотеза, не тикет**.

**Развилка на входе, одна строка:** решено делать, вопрос только «как
доставить» — это **мандат**, скилл `feature-intake`. Не уверены, что взлетит,
и хотите проверить данными — **ставка**, работаем здесь.

`H` (что верим и почему) → `A` (что делаем) → `D` (что меряем) → `I` (что
узнали) → инсайт питает следующую `H`. Метод — `team-roles/references/methodologies.md`.

**Первым ходом** — конфиг проекта (скилл `project-config`):
`files.hypotheses_dir`, `hypothesis_folder`, `epic_folder`, `routes`,
`canon_dir`/`canon_index`, `models.*`, `roles.*`.

## Пять раздельных шагов

Шаги запускаются отдельно и в разное время — скилл не склеивает их в кнопку.

| Шаг | Что происходит | Выход |
|---|---|---|
| **Г1 завести** | сырьё → карточка гипотезы | файл-карточка + дыры вопросами |
| **Г2 челлендж** | другая модель пробует развалить | в очередь / доработка / отклонена с причиной |
| **Г3 развернуть** | постановка работы | эпик + папка + нарезка задач списком |
| **Г4 замер** | снять **заявленные** срезы | файл замера с цифрами и пруфами |
| **Г5 вердикт** | решение + инсайт | запись в канон, вход следующей гипотезы |

### Г1 — завести гипотезу

Сырьё каша → сначала `intake-groom` дешёвой моделью. Дальше — карточка по
`assets/hypothesis-template.md` в `files.hypotheses_dir` (пакет — папка по
шаблону `hypothesis_folder`).

**Нет основания — нет гипотезы.** Основание = цифра из выгрузки, звонок,
жалоба, поведение конкурента — **с пруфом**. Без него это хотелка: из
черновика не выходит.

**Требования по отделам — из `routes` конфига**, только вовлечённые, объединять
где скоуп один. MECE считается по тем блокам, что есть.

### Г2 — челлендж

Прогнать через `cross-review` (другая модель пробует развалить расчёт и логику)
+ сверка с журналом закрытых вопросов проекта: не отвечали ли на это раньше.
Исход — **решает человек**: в очередь / на доработку / отклонена **с причиной**.

Проработка не проходит без двух чисел: **база** (сколько сейчас) и **размер
приза** (сколько на кону). Иначе строится механика ради копеек.

### Г3 — развернуть

Ровно как Р2 у `feature-intake`, переиспользуем и не дублируем: эпик-контейнер
по `routes` → папка эпика по `epic_folder` → сшивка ссылок в обе стороны →
**нарезка задач списком**. Тикеты заводит груминг (`jira-task-standard`),
не скилл.

### Г4 — замер

**Гейт «есть чем мерить»** — перед стартом замера проверить, что события шлются,
дашборд собран, разметка на месте. Нет — вернуть в постановку задачей аналитике,
а не «замерять как получится».

Снимаются **ровно те метрики, что заявлены на Г1** — не подобранные постфактум.
Файл — `assets/measurement-template.md`, в пакет гипотезы.

### Г5 — вердикт

Решение: **масштабировать / итерировать / закрыть** — по данным. И **инсайт**:
что узнали про пользователя или механику, а не «зашло/не зашло». Инсайт
уходит в `canon_index`/журнал проекта и становится основанием следующей
гипотезы. Вопрос закрыт навсегда → стадия «не переоткрывать».

## Стадии и правило владельца

Рабочие: `черновик → проработка → челлендж → отдана → раскатана → замер →
вердикт`. Концовки: `отклонена (с причиной)` · `заморожена` (ждёт данных или
решения) · `не переоткрывать`.

У каждой стадии в шапке карточки — **дата входа**: видно не только «где висит»,
но и «сколько».

**Правило владельца:** со стадии «отдана» и далее обязательны `владелец` и
`следующий_шаг` с датой, иначе стадия не закрывается. Это лечит «висит два
месяца, все думали, что не их».

## Три вещи, которые лечат реальные грабли

1. **Правило владельца** — против зависания.
2. **Гейт «есть чем мерить»** — против «раскатали, а замерить нечем».
3. **Метрики заявлены заранее** — против подбора красиво легших цифр постфактум.

## Синхронизация трёх слоёв

**Канон** (файлы пакета — источник) → **публикация** (Confluence/доки, рендер
канона) → **исполнение** (трекер). Правки идут из канона наружу, не наоборот.

Отдельный **шаг сверки** на стадии «отдана»: пройти по ключам тикетов →
изменения принести обратно в карточку → перепубликовать → строка в лог решений.
У каждого поля один хозяин, остальные — копии с датой. **Расхождения показывать
человеку списком, не править молча.**

## Кто на каких стадиях (роли — скилл `team-roles`)

| Стадия | Роль | Выход |
|---|---|---|
| черновик | `intake-groom` (дешёвая модель) | карточка: JTBD + основание + дыры |
| проработка | `product` + `data-analyst` (база и приз — жёсткий гейт); по надобности `ux-researcher` (поведение) и `market-researcher` (основание извне) | карточка целиком + расчёт |
| челлендж | `cross-review` + журнал закрытых вопросов | вердикт по гипотезе — за человеком |
| отдана | `product` | работа поставлена, ссылки сшиты, владелец и дата есть |
| раскатана | `data-analyst` | гейт «есть чем мерить» пройден |
| замер | `data-analyst` | файл замера с пруфами |
| вердикт | `product` + владелец | решение + инсайт в канон |

Человек-владелец обязателен в трёх точках: **челлендж, постановка, вердикт**.

## Чего скилл НЕ делает

- **Не заводит тикеты** — это груминг.
- **Не подменяет `feature-intake`** — мандат идёт туда.
- **Не считает аналитику сам** — это роль `data-analyst`.
- **Не выдумывает** цифры и условия: дыра → открытый вопрос.

## Смежные скиллы

- `project-config` — пути, отделы, роли, канон.
- `feature-intake` — второй вход (мандат) и общий шаг разворачивания.
- `intake-groom` — черновик из сырья.
- `cross-review` — челлендж на Г2.
- `jira-task-standard` — форма тикетов после Г3.
- `nyron-waves` — доставка.
- `team-roles` + `references/methodologies.md` — кто и каким методом работает.
