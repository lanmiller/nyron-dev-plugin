# feature-intake — требование как артефакт

У продукта есть вход, которого не покрывает ни один другой скилл плагина:
**решённая фича**. Стейкхолдер (продакт, CEO, комплаенс, сам разработчик) хочет
добавить функционал. Это **мандат** — его не проверяют ставкой, его доставляют.

Плагин силён справа (доставка: `nyron-waves`) и в форме тикета
(`jira-task-standard`). Слева не хватало **формы самого требования** и **гида
разворачивания**. Этот скилл — только про них, тонкий: всё остальное делают
смежные скиллы, он на них ссылается и не копирует.

**Развилка на входе, одна строка:** не уверены, что взлетит, и хотите проверить
замером — это **ставка**, ей занимается скилл `product-flow`, не этот. Уверены и
решили делать — **мандат**, работаем здесь.

## Два раздельных шага

Требование и эпик — разные события, между ними живёт обсуждение и согласование
(созвон, комменты, правки). Скилл НЕ склеивает их в одну кнопку.

| | Р1 «Собрать требование» | Р2 «Развернуть в эпик» |
|---|---|---|
| Когда | принесли хотелку | требование согласовано (`status: согласовано`) |
| Кто запускает | автор требования | тот, кто берёт в работу (продакт/диспетчер) |
| Выход | файл-требование + открытые вопросы | эпик в трекере + папка эпика + нарезка задач списком |
| Тикеты задач | не заводит | не заводит (это груминг) |

---

## Р1 — собрать требование

1. **Конфиг**: прочитать конфиг проекта (скилл `project-config`) —
   `storage`, `files.requirements_dir`, `routes`, `models.*`.
2. **Принять сырьё**: текст, голос-расшифровка, скрин, ссылка, пересылка.
   Сырьё — каша (правки поверх правок, несколько хотелок) → сначала
   `intake-groom` дешёвой моделью, дальше работать с его выжимкой.
   Сырьё сохраняется дословно (секция «СЫРЬЁ» в файле) — ничего не теряем.
3. **Довести диалогом**: спрашивать по одному вопросу, не растягивать.
   Обязательный минимум: зачем это бизнесу; кто пользователь; что считаем
   готовым. Дыра в требовании — **открытый вопрос**, не фантазия скилла.
4. **Записать файл** по `assets/requirement-template.md` в
   `files.requirements_dir` (`<слаг>.md`). Фронтматтер:
   `status: draft` · `owner` · `epic: ""` · `tickets: []` · `departments`.
5. **Вернуть человеку**: ссылка на файл + открытые вопросы списком + сигнал
   размера (эпик / задача / мелочь) одной строкой.

**Требования по отделам строятся из `routes` конфига, а не из зашитой
четвёрки.** Включаются ТОЛЬКО вовлечённые отделы: у одного продукта
разработка+дизайн+CRM, у другого один общий скоуп «промо-разработка» —
это нормально, MECE считается по тем блокам, что есть.

Статус `draft → согласовано` снимает **человек**, не скилл. Несогласованное
требование в Р2 не идёт.

## Р2 — развернуть в эпик

Запускается отдельной командой по требованию со `status: согласовано`.

1. **Эпик-контейнер**: создать эпик в трекере — маршрут по `routes` (отдел,
   который ведёт работу), тип эпика и лейблы из конфига трекера. Заголовок =
   глагольная фраза результата, описание = выжимка требования (JTBD + скоуп +
   критерии приёмки), не копия файла.
   `storage: files` → эпика-тикета нет, контейнер = файл-эпик в папке эпика.
2. **Папка эпика** по шаблону `epic_folder` конфига (подстановка `{key}` —
   ключ эпика, `{slug}` — слаг требования). Файл-требование переезжает туда
   (или лежит в `requirements_dir` со ссылкой — как задано в конфиге проекта).
3. **Сшить ссылки в обе стороны**: во фронтматтер требования — `epic: <ключ>`;
   в эпик — ссылка на файл-требование. Тикет без файла и файл без ключа —
   потеря контекста.
4. **Нарезка задач — списком в файле, не тикетами**: MECE-разбиение по
   `routes`/участкам, у каждой строки — результат и отдел. Это черновик для
   груминга.
5. **Передать в доставку**: сказать человеку, что дальше — груминг по
   `jira-task-standard` (форма тикетов) и `nyron-waves` (волны). Скилл
   останавливается здесь.

## Правила

- **Мандат, не ставка.** Никаких HADI-петель, замеров и вердиктов: требование
  считается принятым по DoD, а не по метрике эксперимента.
- **Не выдумывать.** Всё мутное — в «Открытые вопросы» с адресатом. До ответа
  требование остаётся `draft`.
- **Человеческий слой без сленга** (McKinsey: пирамида, кратко, однозначно;
  MECE в критериях приёмки) — канон стиля в `jira-task-standard`, не дублируем.
- **Один артефакт — одна цель.** Две несвязанные хотелки → два требования.
- **Тикеты задач скилл не заводит** никогда: сырые тикеты потом всё равно
  переписывать на груминге.

## Смежные скиллы

- `project-config` — куда писать, какие отделы и типы у трекера.
- `intake-groom` — причесать кашу до Р1 (дёшево, sonnet/haiku).
- `jira-task-standard` — форма тикетов после Р2 и канон стиля.
- `nyron-waves` — доставка эпика волнами.
- `product-flow` — второй продуктовый вход: ставка (гипотеза → замер → вердикт);
  шаг разворачивания в эпик у нас общий.
- `superpowers:brainstorming` — если требование мутное и «хочется вот такого»:
  расшторм с автором ДО записи файла (мягкая деградация: нет плагина — просто
  диалог по шагу 3).
