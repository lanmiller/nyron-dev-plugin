# cross-review — проверка кода другой моделью

Ключевое правило конвейера: **кто писал — тот не проверяет**. Код в проекте
пишет Claude (Opus-субагенты) — значит проверяет GPT. Если однажды код начнёт
писать GPT — проверять его будет Claude. Всегда крест-накрест; самопроверка
той же моделью вердиктом не считается.

Зачем это системе: пока вердикт выносит только человек, он — узкое место, и
волны стоят в очереди на его глаза. Независимый машинный ревьюер позволяет
работе доезжать до «готово» без участия человека; человек смотрит только то,
что ревьюер уже пропустил.

## Механика

Ревьюер — GPT через **codex CLI** (`codex exec`, ChatGPT-подписка, API-ключ не
нужен). Запуск — готовым скриптом:

```bash
"<этот-скилл>/scripts/cross-review.sh" -C <repo-worktree> -b main \
  -t <файл-с-тикетом> -m gpt-5.6
```

- `-C` — worktree проверяемой ветки (HEAD = ветка), `-b` — базовая ветка диффа;
- `-t` — файл с контекстом тикета: JTBD, DoD, «Как тестировать» (сохранить
  описание тикета во временный файл перед запуском; без него ревьюер оценивает
  код в вакууме — хуже). В этот же файл ДОБАВИТЬ секцию «Режимы проекта» со
  списком `flows:` из конфига проекта — по ней ревьюер и план-челлендж
  проверяют смежные узлы (реальные соседние режимы, трогающие те же данные:
  типовой источник багов «спека про создание, а рестарт не отработан»);
- `-m` — модель из конфига проекта `reviewer.model` (скилл `project-config`);
  не задана — дефолт codex.

**Запуск — ФОНОМ, не синхронно**: ревью на Sol длится 2–5+ минут — Bash-вызов
скрипта делать `run_in_background: true` (вывод в файл через `-o`/редирект),
основной поток НЕ блокировать: пилить следующий тикет/шаг, вердикт забрать по
нотификации завершения. Синхронный вызов допустим только когда работы больше
нет и сессия всё равно ждёт. MCP-тул `codex` (блокирующий по природе) — для
коротких вопросов GPT; длинное — только фоновым Bash. В Workflow-скриптах
синхронность внутри agent() не вредит — параллелит сам конвейер.

Скрипт сам собирает `git log`/`git diff` от merge-base, строит промт с жёстким
форматом вердикта и зовёт codex в read-only-песочнице (модель может ЧИТАТЬ
репо, но не менять). Дифф >300KB в промт не кладётся — ревьюер читает файлы
сам. stdout скрипта — вердикт, первая строка строго:

- `ВЕРДИКТ: ПРИНЯТО` — плюс 1-3 строки, что проверено;
- `ВЕРДИКТ: ДОРАБОТКА` — плюс нумерованные замечания
  (файл:строка — что — чем грозит — как чинить); блок «НЕБЛОКИРУЮЩЕЕ:» —
  советы, не влияющие на вердикт.

Технические ошибки скрипта (нет codex, нет коммитов) — ненулевой exit code:
отличать от вердикта, не интерпретировать как ДОРАБОТКУ.

## Встраивание в конвейер волны (nyron-waves)

1. **Волна перед сдачей**: запушила ветку → прогнала cross-review → вердикт
   ПРИЛОЖИЛА к отчёт-комменту тикета (полностью). ДОРАБОТКА от GPT — волна
   чинит ДО перевода тикета в Ревью и гоняет ревью повторно; в Ревью тикет
   уходит только с ПРИНЯТО (или с явной пометкой, почему замечание отвергнуто —
   ревьюер может ошибаться, аргументированный отвод легитимен).
2. **Диспетчер на вердикте**: видит в отчёте вердикт GPT; сомневается —
   перегоняет сам (свежая ветка после доработок). Вердикт диспетчера в тикет
   опирается на: GPT-вердикт + собственную верификацию (`git log`, тесты).
3. **Спор моделей**: Claude-автор не согласен с замечанием GPT — решает
   диспетчер (или человек), несогласие фиксируется комментом; молча
   игнорировать замечания нельзя.

Конфиг проекта `reviewer.engine: none` — шаг пропускается (но это осознанное
решение проекта, не дефолт).

## Границы

- GPT — **ревьюер, не диспетчер и не автор**. За руль (диспетчер) его не
  сажать; писать код им — только когда упёрлись в лимиты Claude, и тогда
  проверяет уже Claude.
- Ревью — read-only: скрипт не даёт codex править файлы. Чинит замечания
  всегда автор (Claude-субагенты).
- Секреты в промт не попадают: скрипт передаёт только дифф и текст тикета.

## Второй контур: челлендж плана ДО имплементации

Для крупных задач (новый контур, >1 репо, эпик-декомпозиция) вторая модель
включается ещё на уровне спеки/раскадровки — двумя головами:

- **Fable (thinker)** — штурм узких мест мышлением: архитектурные развилки,
  скрытая сложность, «а надо ли вообще так»;
- **Sol** — сверка плана с РЕАЛЬНЫМ кодом: `scripts/plan-challenge.sh
  -C <repo> -t <файл-спеки>` (read-only по репо). Ловит ложные предпосылки
  (план ссылается на несуществующее), дубли легаси, пересечения нарезки по
  файлам, дыры DoD, более простой путь. Вердикт: `ПЛАН: ОК` / `ПЛАН: РИСКИ`
  списком с пруфами из кода.

Риски правят ПЛАН до выдачи брифов/чипов — поймать на бумаге стоит копейки.
Мелким задачам (S: баг/одна файл-зона) челлендж не нужен.

## Альтернативы: живой GPT-тул и нативный ревью

- **MCP-сервер `codex` плагина** (`codex mcp-server`) даёт GPT тулом в каждой
  сессии — для живой делегации вне конвейера («посмотри этот файл», «предложи
  альтернативу»). Для вердиктов волн использовать НАШ скрипт — его формат
  машиночитаем для шины Jira.
- `codex exec review --base main` — нативный ревью-режим codex для быстрой
  локальной проверки без нашего формата.
