# Live-тестирование эпика (посекундная методика)

Режим: **пользователь у экрана кликает по продукту, сессия — наблюдатель-
диагност**. Цель — не «проверить, что работает», а снять посекундную правду о
том, как флоу работает НА САМОМ ДЕЛЕ, найти расхождения с ожиданием и
конвертировать каждое в тикет с пруфом. Выход прогона: обновлённая карта флоу
+ пачка тикетов + закрытые live-подтверждениями старые тикеты.

## Подготовка стенда (до первого клика)

1. **Определить объект**: эпик/фича, доска/URL, какие сервисы в цепочке
   (фронт → бек → агент → БД/очереди). Выписать ID: workflow, chat, ключевые
   сущности — по ним фильтровать логи.
2. **Мониторы на ВСЕ слои сразу, персистентные** (не разовые снятия):
   - продуктовый агент: `ssh <агент-сервер> docker logs -f <контейнер>` c
     grep-фильтром по structured-маркерам (тулзы, ходы, ошибки);
   - бек: `ssh <бек-сервер> docker logs -f <контейнер>` по маркерам
     доставки/persist/broadcast;
   - БД: прямые запросы (persisted-правда) — сверка с live;
   - фронт: браузер-пейн второй сессией того же пользователя — читать
     **DOM/innerText и консоль/сеть, НЕ скриншоты** (скриншоты — только для
     визуальных багов).
   Сторожки-watchdog'и на события (until-loop в фоне) — чтобы не поллить руками.
3. **Базлайны до теста**: снять md5/дампы того, что НЕ должно измениться
   (данные, промты, конфиги) — после теста сверить diff'ом, а не глазами.
4. **Не деплоить во время прогона**: пуш в main = перезагрузка стенда
   пользователя. Фиксы копить и катить пачкой после.

## Посекундная разметка (ядро методики)

На каждое видимое событие — строка в хранимой таблице:

| Время | Событие | Слой | Этап карты |
|---|---|---|---|

- **Время** — лог-время источника (не «примерно»); при расхождении слоёв
  фиксировать все точки: агент начал / бек persisted / пользователь увидел.
- **Слой** — кто источник: юзер / фронт / бек / агент / агент(саб) / БД.
- Аномалия (тишина, дубль, пропажа, скачок) → пометка ⚠️ прямо в строке и
  немедленная проверка ВСЕХ источников параллельно — у одного симптома
  бывает 2-3 независимые причины, искать до конца, не останавливаться на первой.

## Карта флоу (хранимый артефакт)

Файл в docs-репо (`ai-evolve-docs-test/…`), структура секции прогона:
1. «Как читать трек» — методика/слои (один раз);
2. описание «как устроено» — этапы флоу, какие агенты/промты/тулзы/файлы
   на каждом (формат «ожидание ↔ факт ↔ чем закрыто»);
3. живые посекундные таблицы по этапам;
4. блок «Сошлось / Находки» со ссылками на тикеты.
Карта = фактическая трасса переходов системы: из неё строится State Transition
диаграмма для блоков «Как тестировать» (см. jira-task-standard). Пушить после
каждого этапа — она нужна и остальной команде.

## Находка = тикет сразу

- Каждое расхождение ожидания и факта — тикет ПО ХОДУ прогона (не «потом
  спишем»), по стандарту `jira-task-standard`, с пруфом: лог-строки, id
  сообщений/сущностей, время. Пруф кладётся и в тикет, и в строку карты.
- Перед созданием — сверка с эпиком на дубль; если старый тикет подтверждён
  прогоном зелёным кейсом — сразу в Готово с пруф-комментом («каждый зелёный
  кейс = тикет в Готово»).
- Ложные тревоги тоже фиксировать (в карту): «показалось X — оказалось Y»
  экономит следующую отладку.

## Дисциплина сессии-наблюдателя

- Не верить одному источнику: чат читать из БД + DOM (live ≠ persisted —
  расхождение само по себе баг двух видов: потеря live-доставки ИЛИ потеря
  персиста; после F5 сверять ещё раз).
- F5-чеки — часть прогона: снять состояние до, перезагрузить, сверить с БД.
- Тайминги мерить и раскладывать: «долго» — не находка; «97с, из них 40с
  ретрай из-за X и 45с тройной вызов Y» — находка с двумя тикетами.
- Пользователь ведёт продукт, сессия НЕ вмешивается в его стенд без просьбы;
  свои клики — только во второй сессии-наблюдателе.
- Очередь прогонов вести явно (№, цель, что добиваем) — прогоны короткие и
  тематические лучше одного бесконечного.

## Выход прогона

1. Карта флоу дописана и запушена; 2. тикеты: новые с пруфами / старые в
Готово; 3. сводка пользователю: что сошлось, что нет, очередь фиксов;
4. если эпик добит и все тикеты закрыты — передача QA по скиллу `qa-handoff`.
