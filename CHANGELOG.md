## 0.5.1 — 2026-07-22

- **Тест-бюджет из DoD** (конвейер, тест-ген): 1 пункт DoD = 1-2 теста +
  негативные из «Как тестировать», MECE; тест, не доказывающий пункт DoD, —
  лишний код (боль юзера: «туча тестов и лишние прогоны»).
- **Правило саморедукции слоёв** (`nyron-waves` телеметрия): слой без
  уникальных находок 2 волны подряд вылетает/сужается до СЛОЖНЫХ; каждый
  прогон обязан называть класс ошибок, который ловит только он. Fable в
  конвейере — судья (синтез сигналов + конструкция), не четвёртый ревьюер.

## 0.5.0 — 2026-07-22

координация агентов, экосистема плагинов). Решения зафиксированы с юзером.

- **Конвейер v2 = канон кодового тикета**: тест-ген из DoD силами **Sol**
  (чужая семья, репо тестов, реализацию не видит) → чекпойнт «тесты=спека» →
  код Opus (red→green, тестовые файлы НЕ правит) → механический дифф-валидатор
  (Haiku) → адверсарка → Sol-ревью + полный прогон. Скелет переписан:
  `examples/ticket-conveyor-workflow.md`.
- **Телеметрия конвейера** — `.nyron-hub/metrics.jsonl`: пруф лестницы,
  находки по источникам (sol|adversarial|red-test|dispatcher), вердикты.
  Первые 2 волны — оба ревью-слоя с разметкой → адверсарку всем/только
  СЛОЖНЫМ решаем по таблице перекрытия, не по ощущению.
- **Роутинг моделей переписан**: Fable = решения (целевая модель диспетчера И
  волн — только при конвейере-Workflow), «одна сессия — одна модель навсегда»
  (KV-кэш + человеческий фактор), Sol в таблице как полноправный слой.
- **cross-review п.7**: переусложнение (вкл. «подстилки» под несуществующие
  случаи) — НЕБЛОКИРУЮЩЕЕ; блокер только дубль механизма. НЕБЛОКИРУЮЩЕЕ
  автособирается в файл долгов — не теряется и не шумит.
- **`references/dev-best-practices.md`** — перенос практик на любой проект
  одним файлом: лестница, TDD-контур, режим прогонов, модель-матрица,
  ревью-слои, телеметрия, чек-лист настройки (граф кода, env-канон).

## 0.4.2 — 2026-07-21

**Лестница до кода** — «самый дешёвый код ненаписанный», но с ограничителем.
Идея из ponytail (87k звёзд); плагин целиком НЕ берём — независимая проверка
(C. Eberhardt, Scott Logic) показала, что семь слов промпта дают тот же эффект,
что весь репозиторий, а бенчмарк гонялся на проекте без дизайн-системы, где
ступень «уже есть в зависимостях» не успевает навредить.

- **`nyron-waves`**: три вопроса в бриф КАЖДОЙ кодовой волны, ответ ДО первой
  строки и в отчёт — нужен ли код вообще · нет ли уже в кодовой базе (**пруф
  grep'ом**) · нельзя ли параметризацией существующего. Это наш канон
  «дорабатывай легаси» как обязательный шаг, а не заповедь. Экономия строк —
  не метрика волны; метрика — меньше возвратов на доработку.
- **`quality_floor` в конфиге** — что минимизации НЕ подлежит (компоненты
  дизайн-системы, флаги фич, тесты с регистрацией, проверки доступа через
  канонический стор). Без этого списка «минимизируй» системно спорит с
  канонами проекта и формально выигрывает, потому что его метрика — LOC.
- **`cross-review`**: пункт 7 «переусложнение» (лишний слой без второго
  потребителя · абстракция на вырост · самописное вместо платформы и уже
  установленных зависимостей) — каждое с пруфом, чем заменяется. И обратное
  правило: не предлагать срезать то, за что проект платит кодом осознанно;
  «меньше строк» само по себе не аргумент.
- Продуктовых скиллов (`feature-intake`, `product-flow`, `jira-task-standard`)
  лестница НЕ касается — там критерий краткости про текст, а не про код.

## 0.4.1 — 2026-07-21

- **Фикс: волны и диспетчер теряли друг друга.** Будка якорилась на корень
  git-репозитория, а проект бывает **зонтиком из независимых репо** (ai-evolve =
  ai-evolve-back + n8n + ai-evolve-docs-test + worktree'ы). Сессия с cwd внутри
  саб-репо получала **приватную пустую будку**: её посты и вотчеры не встречались
  с диспетчерскими, цепочка пробуждения крутилась вхолостую (в прогоне 21.07 —
  три осиротевших `.nyron-hub/.watchers` в саб-репо, `messages.jsonl` в них так
  и не появился). Теперь якорь — **корень проекта**: вверх по дереву до первого
  каталога с `.claude/nyron-dev.md`, затем фолбэк на git-common-dir, затем cwd;
  `NYRON_HUB_DIR` перекрывает всё. Лестница вынесена в `hub/hub-dir.mjs` и
  продублирована в `hub-watch.sh` (правки — только парой).
- **`hub-watch.sh hubdir`** — диагностика: печатает, в какую будку смотрит эта
  сессия. Пути в двух сессиях расходятся → координации нет.
- **`hub/hub-anchor-test.sh`** — тест якоря на синтетическом зонтике (корень,
  два саб-репо, worktree, env-оверрайд, фолбэк). Гоняется без стенда.

## 0.4.0 — 2026-07-21

Вторая половина продуктовой рамки: **ставка**. Теперь два продуктовых входа
(`feature-intake` мандат · `product-flow` ставка) → одна доставка.

- **Новый скилл `product-flow`** — цикл гипотезы как HADI-петля. Пять
  раздельных шагов: Г1 завести (карточка, **нет основания с пруфом — нет
  гипотезы**) · Г2 челлендж (`cross-review` + журнал закрытых вопросов, решает
  человек) · Г3 развернуть (общий с `feature-intake` шаг: эпик + `epic_folder`
  + нарезка списком) · Г4 замер (**гейт «есть чем мерить»**, срезы только
  заявленные на Г1) · Г5 вердикт (**инсайт**, а не «зашло/не зашло» → в канон →
  вход следующей гипотезы). Стадии с датой входа + **правило владельца** со
  стадии «отдана». Синхронизация трёх слоёв: канон → публикация → исполнение,
  расхождения показывать человеком списком, не править молча. Тикеты не заводит.
- **Шаблоны** `assets/hypothesis-template.md` (блоки по пирамиде + MECE, отделы
  из `routes`, план замера заранее) и `assets/measurement-template.md`
  (гейт, таблица «заявлено · до · после · дельта · пруф», вердикт-инсайт).
- **Три новые ролевые карточки**: `ux-researcher` (поведение и джобы, НЕ
  экраны), `market-researcher` (внешний мир, метод `deep-research`, НЕ наш
  пользователь и НЕ наши данные), `marketer` (коммуникации, офферы; условия
  не выдумывает → открытый вопрос).
- **`team-roles/references/methodologies.md`** — HADI · JTBD · Lean ·
  Data-Driven · McKinsey-пирамида + MECE: что это · как применять · чек-вопросы
  роли. Плюс Data-Driven-гейт (без базы и размера эффекта проработка не
  проходит) и правило «вердикт = переиспользуемый инсайт».
- **Конфиг**: `files.hypotheses_dir`, `hypothesis_folder` (`{id}`/`{slug}`),
  `canon_dir` / `canon_index` (память продукта: значение · дата · источник;
  пополняется шагами `product-flow`, а не отдельной рутиной).

## 0.2.16 — 2026-07-20

Пачка надёжности будки (по фидбеку прогона 14 волн):

- **hub/hub-watch.sh — стандартный будка-вотчер** (свои until-loop'ы
  запрещены — источник эхо-багов): фильтр эха и pid-маячок встроены,
  `watch <имя>` — ждать чужое, `alive <имя>` — проверка живости.

- **REARM-FIRST**: перевзвод вотчера ПЕРВЫМ действием после пробуждения,
  до обработки события («перевзвод в конце» — забывается).
- **Чек-лист каждого пробуждения диспетчера** (включая сообщения юзера и
  конец долгих операций): вотчеры живы → hub_status → JQL → остальное.
  Лечит «разрыв/забыл/отвлёкся/после ревью не врубил».
- **Фильтр эха в будка-вотчерах** (live-находка волны: вотчер будился о
  собственный hub_post) — теперь встроен в hub-watch.sh.
- **Реаниматор + коллективная парковка** («упёрлись в 5-час лимиты и чо
  дальше» — полностью БЕЗ человека). Парковка: limit-сигнал (warning/первая
  429) → волна НЕ дожигает окно: WIP-коммит чекпойнта + hub_post «ПАРКУЮСЬ»
  + спать на будке; диспетчер рассылает «ПАРКОВКА ВСЕМ» и паркуется сам;
  takeover по лимитам запрещён. Подъём: `hub-watch.sh reanimator N` —
  zero-token дежурный: тишина N мин → RESUME-пинг напрямую в файл будки →
  будит вотчеры всех сессий; токенов нет — пингует до победного
  (shell-вотчеры переживают лимит, цепочка жива всю паузу). Доказано
  тестом: stale-будка → пинг → вотчер волны проснулся. Профилактика:
  пачки от свежего окна, чипы каскадом, модельная экономика.

превращается из набора delivery-скиллов в переиспользуемую основу для любого

- **`cross-review` (новый скилл + scripts/cross-review.sh)** — кросс-ревью
  другой моделью: GPT через codex CLI (ChatGPT-подписка, read-only песочница)

  перед сдачей в Ревью. Разблокирует «доезжает до готово без глаз человека».
- **`intake-groom` (новый скилл)** — чистильщик входящих: сырая фича →
  draft-тикет по jira-task-standard силами sonnet/haiku + открытые вопросы
  автору; сырьё сохраняется дословно комментом; draft в волны не входит.
- **`team-roles` (новый скилл)** — ролевые карточки: продакт, UX-дизайнер,
  дата-аналитик, техлид, SRE-лид, QA (миссия/вход/выход/чек-лист/границы).
  Generic в плагине, проектное (модель, «куда смотреть», заметки) — из
  конфига `roles.*`. Диспетчер/ревьюер/чистильщик ролями не являются.
- **`project-config` (новый скилл + шаблон)** — конфиг проекта
  `.claude/nyron-dev.md` (+`.local.md` оверрайды): тип development/research,
  хранилище задач jira/files, карта моделей по ролям, движок ревью, роли.
  «Как делать — в скилл, чем проекты отличаются — в конфиг»; один плагин на
  Evolve/Betzo/psylua.
- **`nyron-hub` (новый MCP-сервер, hub/server.mjs)** — «будка»: zero-deps
  Node stdio-сервер в плагине; шина сообщений агентов (hub_post/read), бронь
  файлов с детекцией пересечений (hub_lock/unlock), очередь мержа по репо —
  вливаем по одной от головы (hub_merge_join/leave). Состояние — файлы
  `.nyron-hub/` проекта; Jira будка НЕ трогает (Jira = долгоживущее,
  будка = быстрое служебное). Jira остаётся per-project MCP.
